## WRF and the ARW Dynamical Core
The Advanced Research WRF (ARW) model is a non-hydrostatic, compressible atmospheric
solver formulated on a terrain-following mass coordinate.1 The governing equations are cast in
flux form, utilizing a vertical coordinate defined by the dry hydrostatic pressure of the

atmospheric column.1 The dry hydrostatic coordinate       is normalized and defined as:




In this coordinate formulation,      represents the dry hydrostatic pressure,         is the

the model top.1 The variable      represents the total mass of dry air per unit area within the

Spatial discretization in the ARW core utilizes an Arakawa C-grid staggering scheme.1

Thermodynamic variables—such as potential temperature         , geopotential    , and moisture

mixing ratios    —are positioned at the center of each grid cell, while the velocity components

(          ) are staggered horizontally and vertically on the respective cell faces.1 This spatial
configuration minimizes grid-scale noise and optimizes the representation of pressure gradient
forces and divergence fields.
Temporal integration employs a third-order Runge-Kutta (RK3) time-split scheme.1 This
split-explicit approach integrates high-frequency acoustic modes over smaller substeps to
satisfy the Courant-Friedrichs-Lewy (CFL) stability criterion without incurring the extreme
computational cost of running the complete, slow-variable dynamics core at the acoustic step
frequency.1
To maintain stability and boundary fidelity during regional nest simulations, such as the Canary
Islands 3 km domain replay, the model implements a specified lateral boundary condition (LBC)
coupled with a lateral relaxation zone.2 The outermost grid row and column are directly
specified by external boundary data, while an inner relaxation zone spanning several grid points
applies a Newtonian and diffusion-based damping term to smoothly transition the external
boundary values into the model's interior solution.

## GPU Porting of NWP and Climate Models
The high-performance computing landscape for weather and climate modeling has witnessed
numerous attempts to adapt legacy Fortran bases to GPU architectures.15 These efforts
broadly fall into three categories: compiler directive-based ports, Domain-Specific Language
(DSL) generation systems, and clean-slate C++/Kokkos implementations.17
Directive-based ports of legacy WRF codes—using CUDA Fortran or OpenACC directives
inserted into the original source code—have historically hit a performance ceiling of

approximately     to       speedup.14 This restriction stems from the heterogeneous structure
of WRF, which requires frequent synchronization and data copying between host and device
memory (D2H and H2D transfers) to execute unported physical parameterizations or complex
boundary condition routines on the host. Platforms like TempoQuest’s AceCAST achieved

speedups ranging from             to      by porting select microphysics and dynamics kernels to
NVIDIA GPUs, yet remained fundamentally limited by these CPU-GPU memory transfer
bottlenecks.20
To decouple scientific formulation from hardware-specific optimizations, domain-specific
languages have been introduced. The Pace model (developed by NOAA/GFDL and the Allen
Institute for Artificial Intelligence) ported the Finite-Volume Cubed-Sphere (FV3) dynamical
core to Python using the GT4Py DSL and the DaCe (Data-Centric Parallel Programming)

framework.19 Pace demonstrated            to        speedups relative to highly optimized CPU

DSL-based compilation path with the ICON-exclaim project, leveraging GT4Py to optimize

performance gain on GPU nodes. However, these systems often rely on hybrid execution:
during coupled dynamics-physics runs, unported physics packages force large
multi-dimensional arrays back to the host CPU via MPI or custom bindings, degrading the
end-to-end performance.26
The most prominent clean-slate exascale atmospheric model is the Simple Cloud-Resolving
E3SM Atmosphere Model (SCREAM), written in C++ using the Kokkos programming model to
abstract hardware execution across heterogeneous architectures.16 Running on the Frontier
exascale system, SCREAM achieved an unprecedented throughput of 1.26 simulated years per
day (SYPD) at a global cloud-permitting resolution of 3.25 km utilizing 27,000 AMD MI250
GPUs.18 Another historical precursor is NOAA’s Non-hydrostatic Icosahedral Model (NIM),
which pioneered native GPU execution in 2009 via the custom F2C-ACC Fortran-to-CUDA

compiler, demonstrating up to        speedups on a single GPU core relative to legacy CPU

While numerical models resolve the deterministic Euler equations, machine learning (ML)
emulators have emerged as an alternative forecasting paradigm. DeepMind's GraphCast
operates on an icosahedral grid mapping a multi-scale graph neural network to generate highly
accurate medium-range forecasts.31 Huawei’s Pangu-Weather leverages 3D Earth-specific
transformer architectures to achieve superior skill in meteorological fields.32 Google and
ECMWF introduced NeuralGCM, a hybrid approach combining a highly efficient differentiable
dynamical core (spectral) with ML-based physical parameterization emulators.32 More recently,
the ECMWF Artificial Intelligence Forecasting System (AIFS) has transitioned to operational
status, running on octahedral reduced Gaussian grids.31
The JAX-native architecture presented in this report is positioned not as a competitor to these
ML emulators, but as a complementary, physics-faithful dynamical core. ML models are
fundamentally bounded by their training datasets (e.g., reanalysis fields) and often violate strict
thermodynamic invariants. An ultra-fast, GPU-native numerical core serves as both an
on-demand training-data generator and a physical constraint boundary for hybrid modeling.

## GPU Programming for Scientific Computing
The transition of scientific computations to Python-based implementations has historically
been associated with severe speed penalties due to the interpreted nature of the language.
However, the introduction of JAX—coupled with the Accelerated Linear Algebra (XLA)
compiler—redefines this landscape by transforming Python code into highly optimized, fused
CUDA kernels at runtime.26
JAX utilizes tracing to construct an intermediate representation (Jaxpr) of array
transformations.26 XLA compiles this representation, fusing adjacent stencil operators to
reduce memory bandwidth bottlenecks. This represents a significant advantage over PyTorch,
which is optimized for deep learning and lacks native, automated compile-time fusion for
complex 3D stencil dynamics.
Other modern high-performance frameworks provide alternative paths to GPU execution.
GT4Py acts as a front-end Python DSL that translates stencil computations into
high-performance C++ or CUDA back-ends via DaCe.23 DaCe uses Stateful Dataflow
Multigraphs (SDFGs) to optimize memory layouts and maximize data reuse in L1/L2 caches.19
Kokkos abstracts memory spaces and execution layouts, allowing developers to write
single-source codes that compile to CUDA, ROCm, or OpenMP.18 Triton provides a Python-like
language that enables developers to write highly optimized GPU kernels without directly
writing complex CUDA C code, particularly effective for block-level optimizations.
The primary design pattern driving the current JAX-native implementation of WRF v4 dynamics
is whole-state device residency, which enforces a constitutional invariant of zero in-loop
Device-to-Host (D2H) memory transfers. The entire dry air mass coordinate, staggered velocity
tensors, and moisture arrays remain resident in the RTX 5090's high-speed memory space
throughout the temporal integration, bypassing the high latency of the PCI Express bus.35 The
time loop is formulated using jax.lax.scan, which compiles the entire temporal loop into a single
XLA execution graph, preventing the overhead of launching separate kernels for each timestep.

## AI Agents Building Production Code
The deployment of large language models for software engineering has transitioned from
simple autocomplete features to fully autonomous agentic networks capable of code
generation, structural debugging, and system verification.36
Evaluating LLMs on repository-level code changes requires robust benchmarks. Carlos
Jimenez and co-authors established SWE-bench, an evaluation framework consisting of actual
software engineering problems drawn from GitHub issues across popular Python repositories,
demonstrating the limitations of early models.38 To bridge this gap, John Yang and co-authors
developed SWE-agent, establishing state-of-the-art results through agent-computer
interfaces that allow models to safely interact with an execution environment and iterative test
harnesses.38
Autonomous agent systems have expanded rapidly through open-source developments like
AutoGPT and Microsoft's AutoGen, which facilitate multi-agent conversation frameworks.42
MetaGPT introduced structured roles (such as product manager, architect, and reviewer) to
model software development lifecycles.42 Cognition Labs presented Devin, an autonomous
software engineering assistant equipped with shell access and sandbox tools. Cline and Cursor
integrated these agentic workflows directly into the Integrated Development Environment
(IDE), enabling real-time code refactoring and interactive debugging.
These multi-agent architectures leverage key cognitive patterns described in Anthropic's
agentic guidelines, notably the orchestrator-worker and evaluator-optimizer loops.42 In these
frameworks, a high-level manager agent plans task decomposition and delegates execution to
specialized worker units.42 An independent evaluator agent then critiques the output, driving
iterative self-correction loops until the generated artifact meets predefined quality
thresholds.42 This pattern relies on long-context models, such as Claude 4.7 (1M tokens), to
maintain complete codebase contexts, ADRs, and library dependencies.36
The developed model’s multi-agent swarm architecture was structured specifically to execute
these design patterns:
   ●​ Context-Aware Orchestrator (Claude Opus 4.7, 1M Context Window): Maintained the
       complete software design state, generated per-sprint contract boundaries, and audited
       intermediate implementations.36
   ●​ Specialized Code-LLM Worker (GPT-5.5 Codex): Autonomously generated JAX-native
       stencils and physical parameterization kernels.26
   ●​ Automated Validator Agent (Claude, High-Reasoning Mode): Designed and executed
       unit and integration test fixtures, comparing intermediate outputs against legacy WRF
       savepoints.37
While early agentic attempts have occasionally slipped into generative risks—such as
hallucinating non-existent library APIs or fabricated academic citations—the developed model
enforced a strict contract-driven validation harness to ensure absolute numerical and scientific
integrity.47

## Verification, Validation, and Skill in NWP
The validation of high-resolution atmospheric models bypasses the traditional, often
counterproductive target of achieving strict bitwise parity with legacy Fortran WRF on every
floating-point operation.14 NWP is fundamentally chaotic, and slight variances in compiler
optimization flags, floating-point rounding models, or processor instruction architectures can
trigger large localized grid-point deviations that are meteorologically irrelevant.20
Instead, a robust 4-tier validation pyramid was implemented to verify physical and statistical
consistency 14:
   ●​ Tier 1: Micro Fixture Parity (B-Direct Ladder): Enforces exact, bitwise-identical
       mathematical replication of individual spatial operators against legacy WRF Fortran
       savepoints.14 This is structured as a bottom-up port ladder (Tiers B0 to B6).
   ●​ Tier 2: Physical Invariants: Enforces dry hydrostatic mass conservation (

                            ) within a closed domain, potential temperature physical envelopes

     (         ), and strict non-finite checks.
  ●​ Tier 3: Short-Run Convergence: Verifies that the difference trajectory between the
     legacy model and the GPU rewrite scales predictably according to spatial grid spacing,
     ensuring that discretization errors do not accumulate exponentially over 9-timestep runs.
   ●​ Tier 4: Probabilistic Ensemble Consistency (The Operational Gate): Projects
       standardized outputs of test runs onto the principal components of a perturbed
       reference ensemble using the PyCECT statistical consistency framework.53
To measure spatial forecasting skill without falling into the "double-penalty" trap of point-wise
matching, the model implements the Fractions Skill Score (FSS).54 The FSS compares the
fractional coverage of a target weather event (such as precipitation exceeding a specified

## Regional Weather Forecasting and the Canary
Islands Case
The Canary Islands present an exceptional computational testbed for high-resolution NWP.

Located in the subtropical Atlantic ocean (                     ), the archipelago is characterized
by a complex, steep volcanic orography—including Mt. Teide, which rises abruptly to an
elevation of 3,718 meters over a horizontal distance of less than 15 kilometers.60
The atmospheric environment of this domain is dominated by three highly coupled
meteorological processes:
   ●​ The Trade Wind Inversion: High-pressure cell circulation forces a steady northeasterly
       trade wind flow, which interacts with the steep topography to produce complex
       mechanical waves, localized wind shear, and island-scale wake zones.60
   ●​ Saharan Dust Advection (Calima): Periodic transport of warm, extremely dry Saharan
       air masses aloft (the Saharan Air Layer, or SAL) overrides the cool, moist marine
       boundary layer, creating strong vertical shear and intense thermal inversions.60
   ●​ Orographic Precipitation: Localized condensation and convective development are
       triggered by air masses ascending the windward slopes of the islands, requiring
       high-resolution, double-moment microphysics parameterization to accurately resolve
       precipitation structures.60
To evaluate operational forecast skill, the JAX-native model is continuously scored against the
official Spanish State Meteorological Agency (AEMET) telemetry network.60 Surface station
data is ingested in real time across 1,747 active stations, providing point observations of

2-meter temperature (       ), 10-meter wind components (             ), and total precipitation.60
The model’s forecast values are extracted at the nearest grid coordinate matching the physical
latitude, longitude, and elevation of each station, and evaluated using standard continuous
error metrics (MAE, RMSE, BIAS).62

## Hardware Context: Single-Workstation NWP
The core economic and architectural thesis of this project is that high-resolution regional NWP
can be transitioned from expensive, multi-node supercomputing clusters to a single,
energy-efficient GPU workstation.35 The target deployment machine is configured with a single
consumer-class NVIDIA GeForce RTX 5090 GPU, built on the Blackwell architecture (sm_120),
and equipped with 32 GB of GDDR7 memory.35
The Blackwell memory hierarchy provides a critical advantage over previous architectures.
While high-performance server-grade GPUs (such as the H100 or B200) utilize high-bandwidth
Stacked Memory (HBM3e), consumer-class workstation GPUs utilize GDDR7 memory.35 This
introduces a critical architectural constraint: while compute density is exceptionally high, the
relative memory bandwidth-to-flop ratio is lower. To prevent memory starvation across the
21,760 CUDA cores, the JAX compiler was instructed to enforce strict register and shared
memory utilization 29:
   ●​ Thread Block Structuring: Dynamic grid allocation mapping ensures that column
       physics kernels (such as the Thompson microphysics) are launched with exactly 96
       threads per block, matching a multiple of the Blackwell 32-thread warp size.29 This
       optimizes thread occupancy and memory coalescing.
   ●​ Shared Memory Tiling: 3D stencil operations utilize 2D horizontal tiling structures loaded
       into the high-speed L1 cache/shared memory space, drastically reducing repetitive,
       high-latency global memory fetches.29

## Reproducibility and Open Science in Atmospheric
Modeling
To conform with the FAIR (Findable, Accessible, Interoperable, Reusable) scientific data
guidelines, the JAX-native WRF model codebase and associated verification scripts are made
completely open-source. Legacy models have historically been hindered by restrictive
compilation requirements, proprietary software dependencies, and complex hardware
configuration paths.
By writing the system entirely in Python and JAX, the model runs out of the box in
containerized Docker environments on any system supporting the NVIDIA Container Toolkit.
This eliminates the "compilation hell" of traditional atmospheric model deployment, allowing
researchers to reproduce forecasts of extreme weather events directly on regional
workstations.

      numerical drift in highly stratified regions. Localized physics updates (such as

      single precision (FP32) or mixed half-precision formats to maximize compute throughput
      on the Blackwell Tensor Cores.35
  ●​ ADR-02: Column-Parallel Column Coupling: To minimize global thread synchronization
      overhead, physics-dynamics coupling is performed in a unified vertical column kernel,
       keeping vertical profiles entirely resident within registers and L1 cache during coupled
       updates.
