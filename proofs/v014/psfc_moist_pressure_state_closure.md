## Verdict: FIXED (PSFC diagnostic, WRF-exact) + FORMALLY BOUNDED (3D P state)

The `~210 Pa` vapor-light `PSFC` floor was a **wrong diagnostic formula**, not
a writer artifact and not (directly) the dry pressure state. WRF's runtime
`PSFC` never reads the nonhydrostatic total pressure; it is the **moist
hydrostatic** column integral of the dry-mass coordinate. The GPU computed a
height extrapolation of `P+PB` instead. Production now implements the exact
WRF path; expected post-fix h1 `PSFC` RMSE is **57.8 Pa** (gate: <= 120 Pa).

## 1. WRF source anchoring (exact runtime PSFC path)

Pristine WRFv4 tree `/home/enric/src/wrf_pristine/WRF` (the contract's
`/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF` does not exist
on this box):

1. `phys/module_surface_driver.F:1988` — `PSFC(I,J) = p8w(I,kts,J)`.
2. `dyn_em/module_first_rk_step_part1.F:1400` — the surface driver's `p8w`
   argument is **`grid%p_hyd_w`**, NOT phy_prep's extrapolated `p8w`.
3. `dyn_em/module_big_step_utilities_em.F:4946-4958` (`phy_prep`):

   ```fortran
   p_hyd_w(i,kte,j) = p_top
   qtot = sum over PARAM_FIRST_SCALAR..n_moist of moist(i,k,j,n)
   p_hyd_w(i,k,j) = p_hyd_w(i,k+1,j) - (1.+qtot)*(c1(k)*MUT(i,j)+c2(k))*dnw(k)
   ```

So `PSFC = p_top + sum_k (1+qtot_k)*(c1h_k*(MU+MUB)+c2h_k)*(-dnw_k)`, with
qtot over ALL moist species (qv,qc,qr,qi,qs,qg here).

**Formula proof on CPU truth** (CPU `PSFC` minus the integral recomputed from
CPU `MU/MUB/C1H/C2H/DNW/moist`): RMSE **0.17/0.06/0.06/0.05 Pa** at
h1/h4/h10/h24 (max_abs <= 0.46 Pa; fp32 storage + half-step skew). All-species
beats qv-only at h10/h24 — confirming the `n_moist`
loop. This also RESOLVES the GPT "~14 Pa CPU formula gap" risk: that gap
belonged to the height-extrapolation diagnostic (`PSFC - extrap` = +14.3 Pa),
which is simply not WRF's PSFC path.

## 3. 3D pressure-state characterization (the deeper finding, bounded)

`P+PB` at the lowest mass level vs each run's own hydrostatic column:

| Lead | CPU vs own MOIST p_hyd | GPU vs own MOIST p_hyd | GPU vs own DRY p_hyd |
|---:|---:|---:|---:|
| h1  | -13.5 | -202.7 | **-8.2** |
| h10 | -13.4 | -214.6 | **-14.2** |

CPU WRF's nonhydrostatic pressure rides its moist hydrostatic column; the GPU
pressure rides its **dry** column. Root cause (code anchor): the operational
acoustic w-equation uses the deliberately dry WRF specialization —
`operational_mode.py:118` imports and `:1209/:1274/:1483/:1657` use
`dry_cqw`/`pg_buoy_w_dry` (`dynamics/core/advance_w.py`), i.e. `cq1=1, cq2=0`,
omitting WRF `calc_cq` (`module_big_step_utilities_em.F:787-899`,
`cqw=0.5*qtot` k/k-1 average) and the moist `pg_buoy_w` (`:2419-2500`) vapor
load `-cq2*(c1f*mub+c2f)` + PGF scale `cq1=1/(1+qtot)`. Without the vapor
weight the acoustic solver relaxes the full pressure to dry-hydrostatic
balance within the first hour. `advance_w_wrf`/`acoustic_substep_core` already
accept a `cqw` field (`core/acoustic.py:707`), so the moist fix is a wiring +
re-validation task — but it changes prognostic `P/PH/W` evolution, needs the
acoustic savepoint oracles + a GPU gate, and is NOT bundled into this
diagnostic fix. It remains the explanation for the residual 3D `P` lane
(h10 comparator `P` RMSE 65.7 Pa, k0 bias ~-212 Pa).

## 4. Production patch (WRF-faithful, not a clamp)

- `src/gpuwrf/runtime/operational_mode.py` — `_psfc_from_state(state, metrics)`
  now computes the WRF `p_hyd_w(kts)` moist hydrostatic integral from
  `mu_total`, all six moisture leaves, and resident `DycoreMetrics`
  (`c1h/c2h/dnw/p_top`); called with `namelist.metrics` from
  `compute_m9_diagnostics` (M9 -> wrfout `PSFC`). Constant memory per column,
  inside the existing jitted M9 snapshot, no timestep-loop transfers.
- `src/gpuwrf/io/wrfout_writer.py` — the `PSFC` fallback (`_psfc_default`,
  used only when M9/state PSFC is absent) evaluates the same integral when
  `grid.metrics` is resident; synthetic/test states without metrics keep the
  old height extrapolation.
- `scripts/diag/d03_pressure_knockout.py` — updated to the new 2-arg
  signature (direct consumer).
- `MU/MUB/P/PH/U/V/T/QVAPOR` are untouched by construction (PSFC is
  diagnostic-only output).

Functional check on real run data: the patched production function on GPU h1
fields reproduces the ablation exactly (bias +51.553, RMSE 57.823 Pa) and on
CPU fields reproduces CPU `PSFC` to 0.175 Pa RMSE.
