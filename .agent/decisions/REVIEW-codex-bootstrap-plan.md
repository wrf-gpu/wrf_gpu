## Decision
Accept with required fixes

## Top Three Structural Concerns

1. M1 is simultaneously a hard gate and a soft cap: `PROJECT_PLAN.md` requires a Canary WRF-derived fixture to exit M1, then allows M2 to proceed with placeholder Canary data if four fixture sprints run out (`PROJECT_PLAN.md:102`, `PROJECT_PLAN.md:141`; `MILESTONES.md:15-18`).
2. The M2 bakeoff list drops CUDA Tile / explicit CUDA C++ even though the existing architecture and milestone docs list CUDA Tile as a candidate path (`ARCHITECTURE_PRINCIPLES.md:3-4`; `MILESTONES.md:23-27`; `PROJECT_PLAN.md:58-65`).
3. Canary operational v0 is under-scoped: initial/boundary conditions, map projection, terrain/geog ingestion, surface/land coupling, and concrete NetCDF/restart contracts are not proof objects until vague M7 wording (`PROJECT_SPEC.md:5`, `INTERFACE_CONTRACTS.md:7`, `INTERFACE_CONTRACTS.md:19`, `INTERFACE_CONTRACTS.md:27`, `.agent/milestones/ROADMAP.md:78-85`).

## Findings

1. **severity: blocker**; **artifact + citation:** `PROJECT_PLAN.md` §7/§10 (`PROJECT_PLAN.md:102`, `PROJECT_PLAN.md:141`); **issue:** The plan says M1 cannot exit without one analytic stencil fixture, one analytic column fixture, one Canary WRF-derived fixture, schema, storage policy, and comparison command, but the risk mitigation says M2 may proceed after four fixture sprints "even with placeholder Canary data." That bypasses the proof object gate for M1. **required fix:** Remove the placeholder bypass, or explicitly split "M2 may start" from "M1 is closed" and require human approval plus a documented blocker if the Canary fixture is missing. **evidence:** M1 requires at least one idealized and one Canary fixture (`MILESTONES.md:15-18`); no work may be marked done without a proof object (`AGENTS.md:15-20`; `.agent/rules/non-negotiables.md:3-5`).

2. **severity: major**; **artifact + citation:** `PROJECT_PLAN.md` §5 (`PROJECT_PLAN.md:58-65`), `.agent/milestones/ROADMAP.md` M2 (`.agent/milestones/ROADMAP.md:29-36`); **issue:** The M2 candidate set excludes CUDA Tile / explicit CUDA C++ without explanation, while retaining lower-level Python escape hatches. This silently changes the bootstrap candidate universe. **required fix:** Include a CUDA Tile / explicit CUDA C++ candidate, or write a reviewed, human-approved exclusion rationale before M2 dispatch. **evidence:** Existing architecture principles list "CUDA Tile" and explicit CUDA/CUDA Fortran when justified (`ARCHITECTURE_PRINCIPLES.md:3-4`); the bootstrap milestone lists "CuPy/Numba/CUDA Tile spike" (`MILESTONES.md:23-27`); the performance schema already anticipates `cuda-tile` (`PERFORMANCE_TARGETS.md:24-37`).

3. **severity: major**; **artifact + citation:** `PROJECT_PLAN.md` §5 (`PROJECT_PLAN.md:65`, `PROJECT_PLAN.md:145-150`); **issue:** Excluding CUDA Fortran may be defensible, but the plan turns a research-brief dissent into a pre-M2 human decision instead of an evidence note for the bakeoff. That risks pre-judging a backend family before the M2 evidence process. **required fix:** Reframe CUDA Fortran as "excluded unless a scout produces a concrete NVIDIA-only v0 justification," or require the human decision to cite the portability tradeoff and the bakeoff coverage it is accepting. **evidence:** Architecture principles allow explicit CUDA/CUDA Fortran only if justified (`ARCHITECTURE_PRINCIPLES.md:3-5`); the GPT-5.5 brief treats CUDA Fortran as a tactical NVIDIA-only Plan B, not as disproven (`wrf to gpu gpt5.5 deep research.pdf`, p. 4).

4. **severity: major**; **artifact + citation:** `PROJECT_PLAN.md` §6 (`PROJECT_PLAN.md:82-91`); **issue:** The plan says it "binds the tooling choices" and requires Serialbox-style fixtures, PyCECT-style PCA, and probtest-style tolerance derivation. That hardens tools before M1/M6 evidence and before the project knows which tooling can be integrated. **required fix:** Change the table to "tooling families / acceptance behavior," add a short M1/M6 scout or ADR trigger for exact tools, and keep the proof object independent of PyCECT/probtest implementation names. **evidence:** The validation strategy only requires a probtest/PyCECT-style method (`VALIDATION_STRATEGY.md:17-19`), and the research-input summary lists these as tools to consider (`docs/research-inputs.md:22-27`), not locked dependencies.

5. **severity: major**; **artifact + citation:** `PROJECT_PLAN.md` §6 and ROADMAP M7 (`PROJECT_PLAN.md:91`, `.agent/milestones/ROADMAP.md:74`, `.agent/milestones/ROADMAP.md:83`); **issue:** Full PyCECT on a 100-member CPU ensemble is promoted to an M7 proof object without an explicit cost, data-management, or runtime risk. **required fix:** Add a risk and a staged gate: small ensemble prototype at M6, full ensemble only after storage/runtime budget and baseline-run procedure are approved. **evidence:** Deepthink proposes a 100-member CPU ensemble (`v2 ai driven from scratch plan by deepthink.txt:55`); the GPT-5.5 brief recommends probabilistic and ensemble-style checks as building blocks, not a costed M7 mandate (`wrf to gpu gpt5.5 deep research.pdf`, p. 5).

6. **severity: major**; **artifact + citation:** `PROJECT_PLAN.md` §7 (`PROJECT_PLAN.md:98-105`); **issue:** "Overrides per-milestone files where stricter" creates competing authoritative gates instead of updating the milestone files through the normal repository process. **required fix:** Patch the per-milestone files to match the approved plan, or change this language to "proposed stricter gates pending milestone-file patch." **evidence:** The plan itself says governance files remain authoritative (`PROJECT_PLAN.md:5-6`); repository rules require sprint/milestone proof objects and avoid stale duplicated instructions (`MILESTONES.md:3`; `.agent/rules/documentation-minimalism.md:3-6`).

"Input ingest" is too broad for a WRF-compatible regional system. **required fix:** Add an IC/BC mapping proof object covering source datasets, update cadence, boundary-field variables, interpolation policy, and restart interaction before M7 implementation begins. **evidence:** The v0 target is a WRF-compatible regional forecast path (`PROJECT_SPEC.md:3-5`); `DycoreStepInput` includes boundary conditions (`INTERFACE_CONTRACTS.md:25-27`); WRF compatibility includes useful interfaces, variables, and namelist mapping (`PROJECT_CONSTITUTION.md:11-13`).

## Dissent

- I dissent from the M2 candidate list as written because it omits CUDA Tile / explicit CUDA C++ without resolving the contradiction with existing architecture and milestone docs (`ARCHITECTURE_PRINCIPLES.md:3-4`; `MILESTONES.md:23-27`; `PROJECT_PLAN.md:58-65`).
- I do not reject excluding CUDA Fortran from v0, but I dissent from treating that exclusion as settled without a cited human decision or a scoped scout rationale (`PROJECT_PLAN.md:65`, `PROJECT_PLAN.md:145-150`; `wrf to gpu gpt5.5 deep research.pdf`, p. 4).
- I dissent from binding both PyCECT-style PCA and probtest-style derivation as mandatory named tooling before the validation implementation evidence exists (`PROJECT_PLAN.md:82-91`; `VALIDATION_STRATEGY.md:17-19`; `docs/research-inputs.md:22-27`).
- I dissent from allowing M2 to proceed with placeholder Canary data while calling M1 closed (`PROJECT_PLAN.md:102`, `PROJECT_PLAN.md:141`; `MILESTONES.md:15-18`).
- I dissent from deferring METplus research until just before M7 implementation; the observation sources, score definitions, and data-access constraints should be scouted before M6 closeout because M7's pass/fail depends on them (`.agent/milestones/ROADMAP.md:78-86`, `.agent/milestones/ROADMAP.md:104-109`; `wrf to gpu gpt5.5 deep research.pdf`, p. 5).

## Closing

Do not dispatch S1 until the manager removes the M1 placeholder-data bypass, reconciles the M2 candidate set with the architecture principles, and freezes the S1 schema/CLI/storage decisions tightly enough that S2 and M2 can consume the artifacts without reinterpretation. After those edits, patch the plan/roadmap/sprint contract consistently or preserve manager dissent explicitly, then send the unresolved CUDA Fortran and fixture-storage/baseline-source decisions to the human arbiter before any worker touches implementation files.
