## 0. What v0.4.0 builds and why

Replace WRF `real.exe` (`dyn_em/module_initialize_real.F` + `main/real_em.F`)
with a native module that consumes the FROZEN v0.3.0 metgrid-equivalent artifact
(`gpuwrf.init.metgrid_schema.MetEmArtifact`) and produces, with **no CPU-WRF
dependency**:

* a `wrfinput`-equivalent initial `State` + `BaseState` + static `DycoreMetrics`
  (the dynamics half = S1; the surface/soil half = S2), and
* a `wrfbdy`-equivalent `BoundaryState` of per-side specified values + tendencies
  over the forcing intervals (the LBC = S3).

This removes the last external program from the standalone path: v0.3.0 made the
ingest native (raw AIFS + static geog ‚Üí met_em-equiv), v0.4.0 makes the init
native, so the whole pipeline runs on hardware we own and multi-season validation
becomes a compute problem, not a calendar problem.

## 3. FROZEN internal interfaces (the parallel-collision firewall)

New package `src/gpuwrf/init/real_init/` (this commit). The frozen handoff TYPES
are in `real_init/types.py` (`REAL_INIT_TYPES_VERSION = 0.4.0-S0-frozen-2026-06-02`):

* `RealInitConfig` ‚Äî namelist-equiv scalars (nz, p_top, hybrid_opt, etac, eta,
  base-state constants, num_soil_layers, spec_bdy_width, interval_seconds);
  loud-rejects unsupported `hybrid_opt` (P2-2 pattern). Pinned WRF constants
  (G/R_D/CP/RVOVRD/P1000MB/CVPM/...) so every lane uses identical values.
* `VerticalCoord1D` (S1 out), `BaseStateColumns` (S1 out), `DynamicsInit`
  (S1 out), `SurfaceInit` (S2 out), `SoilInit` (S2 out), `LateralBC` (S3 out),
  `RealInitProduct` (the driver's bundle the comparator + runtime-pack consume).
* `WRFINPUT_TOLS` / `WRFBDY_TOLS` ‚Äî predeclared per-field tolerances (section 5).

The lanes are **pure functions producing typed numpy/jax intermediates**, NOT the
GPU-resident `State`. The manager-owned `driver.py` does the single final pack
into `State`/`BaseState`/`DycoreMetrics`/`BoundaryState` (the shared core).
Rationale: keep the manager-owned shared-core `State` contract OUT of the four
lanes ‚Äî they never touch it, so no two lanes (and no lane + the manager) edit the
same file. The frozen entry signatures (already wired in `driver.py`, bodies are
`NotImplementedError` stubs):

```
vertical_coord.compute_vertical_coord(config) -> VerticalCoord1D
base_state.compute_base_state(config, vcoord, hgt) -> BaseStateColumns
vinterp.vertical_interpolate(config, vcoord, metem) -> DynamicsInit   # pre-balance seed
hydrostatic.balance(config, vcoord, base, dyn_seed) -> DynamicsInit   # final P/PH/AL
surface_init.compute_surface_init(config, metem) -> SurfaceInit
soil_init.compute_soil_init(config, metem, surface) -> SoilInit
lateral_bc.generate_lateral_bc(config, forcing_sequence) -> LateralBC
comparator.compare_wrfinput(product, oracle_path) -> CaseParityResult
comparator.compare_wrfbdy(product, oracle_path) -> CaseParityResult
```

**FREEZE POLICY:** no lane may change `types.py` or `driver.py`, or any frozen
signature, without a manager sign-off recorded in this doc + a bump of
`REAL_INIT_TYPES_VERSION`. Additive optional fields on a handoff dataclass are
allowed without a bump if they don't change an existing field.

**Single-source terrain rule (load-bearing):** S1's base state and S2's surface
BOTH need terrain `ht`. The driver sources it ONCE from met_em `HGT_M` and passes
the identical array to both (`compute_base_state(..., surface.hgt)`). A terrain
mismatch between the two lanes would make MUB/PHB inconsistent with the surface
fields and poison hour-0 ‚Äî the freeze forbids each lane deriving its own ht.

**fp64 rule:** real-init runs in fp64 (the hydrostatic balance is precision-
sensitive; the project's fp64-defeat lessons apply). The driver casts to the
operational precision only at the final `State` pack.

## 5. PREDECLARED oracles + tolerances + PASS criteria (NO post-hoc loosening)

Tolerances are frozen in `real_init/types.py` (`WRFINPUT_TOLS`, `WRFBDY_TOLS`).
Metric = RMSE over the domain (with masking) AND a max-abs cap; a field PASSES if
`rmse <= rmse_tol AND maxabs <= maxabs_tol`. Stricter for hour-0-critical dynamics
(MU/MUB/PB/P/PH/PHB/T/U/V), exact for categoricals (XLAND/ISLTYP/IVGTYP), looser
for the 2‚Üí4-layer soil interp (TSLB/SMOIS). Highlights:

| Field | rmse_tol | maxabs_tol | unit | rationale |
|---|---|---|---|---|
| MU / MUB / PB | 5.0 | 50.0 | Pa | dry-mass / base-pressure foundation |
| P | 20.0 | 200.0 | Pa | perturbation pressure |
| PH / PHB | 1.0 | 20.0 | m¬≤s‚Åª¬≤ | geopotential (hour-0 height) |
| T | 0.10 | 1.0 | K | perturbation theta |
| U / V | 0.15 | 1.5 | m s‚Åª¬π | wind |
| QVAPOR | 1e-4 | 1e-3 | kg kg‚Åª¬π | moisture |
| ZNW/ZNU/C1H/C3H | 1e-6 | 1e-5 | ‚Äì | 1D coord near-exact |
| F / E | 1e-9 | 1e-8 | s‚Åª¬π | Coriolis |
| MAPFAC_* | 1e-5 | 1e-4 | ‚Äì | metric passthrough |
| HGT | 0.5 | 5.0 | m | terrain (matches met_em HGT_M tol) |
| TSK / SST | 0.30 | 3.0 | K | skin/SST (protected) |

**Masking policy:** soil/SST fields compared over their valid mask only (soil:
land cells; SST/skin-over-water: water cells); below-ground / fill-value points
excluded; categorical fields compared exactly on all points.

**PASS criteria, predeclared:**
* **S1 gate:** wrfinput ZNU/ZNW/C*/PB/MUB/PHB/P/PH/T/U/V/QVAPOR within tol for

* **S3 gate:** wrfbdy coupled values + tendencies (U/V/T/PH/QV/MU) within tol for

* **S4 gate (closes the milestone):** the comparator runs all of the above as ONE

## 6. Parallelization plan (PARALLEL-BY-DEFAULT) + forced-serial points

Applying the skill's PARALLEL-BY-DEFAULT rule: decompose, then run every
file-disjoint / static-oracle-independent lane concurrently.

**S1 ‚à• S2 ‚à• S3 ‚à• S4 can ALL start concurrently after this S0 freeze.** Each lane
develops against its STATIC oracle (the real.exe wrfinput/wrfbdy fixtures on disk
+ the frozen `MetEmArtifact` schema), NOT against a sibling lane's unmerged output:

* **S1** ‚üÇ S2 ‚üÇ S3: file-disjoint, validate each against its own wrfinput/wrfbdy
  oracle subset.
* **S3's apparent dependency on S1/S2 is NOT a real serialization.** S3 needs
  MU/MUB/MSF at each forcing time to couple ‚Äî but during development it reads
  those from the per-interval oracle wrfinput (a static fixture), so it builds in
  parallel and only at INTEGRATION calls the real S1/S2 entry points (whose
  *interfaces* are frozen here). This is the skill's "develop against the static
  oracle, not the sibling's not-yet-merged output" pattern.
* **S4** builds the comparator + campaign runner against the frozen tol tables +
  the oracle files immediately; it does not need any lane implementation to write
  the harness.

**The ONLY forced-serial points (the justified serialization):**
1. **S5 integration** (manager): wire the four lanes in `driver.py`, pack into
   `State`/`BaseState`/`DycoreMetrics`/`BoundaryState`, resolve any
   single-source-terrain / precision edge. Serial because it touches the
   manager-owned shared core and consumes all four lanes' merged output.
2. **The 24h native-init FORECAST gate (GPU-bound).** ONE GPU job at a time
   (single GPU shared with the v0.3.0 S5 lane + the per-case pipeline). This
   serializes onto the GPU and runs in S5, not in the parallel S4 lane. S4's
   offline field-parity campaign (CPU, cores 0-3) is NOT GPU-bound and runs in
   parallel; only the forecast step waits for the GPU.
3. **v0.3.0 S5** (native met_em production) is a separate lineage; v0.4.0 can use
   the existing real-WPS met_em + the `wps_cases` met_em as the input oracle until
   v0.3.0's native met_em lands. So v0.4.0 does NOT serialize behind v0.3.0 S5 ‚Äî
   they proceed in parallel, joining only at the final fully-native run.

**Recommended 4-wide dispatch (interchange Opus‚ÜîGPT, equal-frequency on the hard
parts; in-process background Opus agents + a tmux codex GPT lane):**

| Lane | Model | Why |
|---|---|---|

## 7. Risks / unresolved

* **No analytic oracle** ‚Äî the entire milestone leans on real.exe fixtures. The
  24h forecast gate is the only end-to-end falsifiable check; everything else is
  field-parity vs real.exe. Mitigation: tolerances frozen up front, masking
  predeclared, structural isolation so a bad lane can't silently leak.
* **hybrid_opt=2 coefficient exactness** ‚Äî the Klemp polynomial branch + the
  `etac` value must match real.exe exactly or PB/PHB drift. `etac` is a namelist
  scalar; S0 leaves it in `RealInitConfig` ‚Äî **S1 must read the actual `etac`
  from the case namelist / wrfinput global attr at impl** (the recon confirmed
  HYBRID_OPT=2 but did not pin etac). TRACKED for S1.
* **2‚Üí4 soil-layer interpolation** ‚Äî `module_soil_pre` exact node depths for
  Noah-MP (sf_surface_physics=4); S2 confirms ZS/DZS against the wrfinput oracle.
* **moist/UM/theta-level branches** ‚Äî real.exe has many `flag_ptheta/flag_prho`
  and AFWA/UM paths the Canary AIFS case does not hit; v0.4.0 supports the
  isobaric-metgrid path only and must loud-reject the others (P2-2 pattern).
* **Single GPU** ‚Äî the forecast gate cannot run concurrently with the v0.3.0 S5
  GPU lane or the per-case pipeline; the manager must schedule one GPU job at a
  time.
