# ADR-009 - RRTMG JAX Implementation Mapping

Date: 2026-05-21
Author: M5-S3 worker draft (Codex gpt-5.5)
Status: PROPOSED worker draft, pending mandatory Claude Opus 4.7 reviewer pass
Scope: post-hoc implementation record for the M5-S3 RRTMG shortwave and longwave radiation column kernels.

## Decision

Decision: keep the M5-S3 RRTMG column candidate as a compact JAX-resident radiation kernel with separate fused shortwave and longwave entry points, device-resident spectral table arrays, Tier-1 fixture parity against a linked source-derived Fortran harness, and Tier-2 radiation invariants. The public APIs are `solve_rrtmg_sw_column(state, tables=RRTMG_TABLES, *, debug=False)` and `solve_rrtmg_lw_column(state, tables=RRTMG_TABLES, *, debug=False)`. The table bundle is a pytree of JAX array leaves, so the spectral coefficients are passed to the compiled function as array arguments rather than captured as Python literals.

This ADR records an implementation candidate, not a claim of full WRF RRTMG equivalence. The harness links the available real WRF RRTMG SW/LW objects and calls their init routines with table file reading disabled, but the per-column oracle remains a compact source-derived two-stream model. That is a named anti-tautology gap carried into the reviewer and manager decision. The gate status is therefore `GO_CARRYFORWARD`, not strict `GO`.

## WRF Source Mapping

The available WRF tree contains `module_ra_rrtmg_sw.F.o` and `module_ra_rrtmg_lw.F.o` under the Gen2 GNU build. The sprint prompt stated "32 bands", but this WRF implementation declares 14 shortwave bands (`nbndsw = 14`, corresponding to RRTMG bands 16-29) and 16 longwave bands (`nbndlw = 16`), for 30 WRF spectral bands in this build. The table asset and constants follow the discovered WRF source rather than padding to a non-existent 16-band SW table.

The full WRF driver signatures are intentionally not reproduced in JAX in this sprint. WRF `RRTMG_SWRAD` and `RRTMG_LWRAD` are grid-driver routines with large optional argument surfaces for cloud overlap, CAM gases, aerosols, and diagnostics. The M5-S3 candidate starts at the column-radiative-transfer level: precomputed `coszen`, layer state, hydrometeors, cloud fraction, surface albedo or emissivity, density, and layer depth. Full WRF driver binding, including exact CAM gas and aerosol option behavior, remains follow-on work.

## Tables

`scripts/extract_rrtmg_tables.py` writes `data/fixtures/rrtmg-tables-v1.npz` with compact SW and LW band weights, gas absorption coefficients, Rayleigh coefficients, and cloud optical coefficients. The extractor records the SHA-256 digests of the WRF source files and the local `RRTMG_SW_DATA` / `RRTMG_LW_DATA` files in a JSON sidecar. This is not a byte-for-byte dump of all private WRF `k` arrays; it is a deterministic M5 column table bundle sized for the first JAX kernel and auditably tied to the local WRF data availability.

The production loader in `rrtmg_tables.py` exposes an `RRTMGTableBundle` named tuple of JAX arrays. SW and LW kernels use `jnp.take` against those arrays in the traced body. This follows the M5-S1.x lesson: spectral arrays are runtime leaves and should not become JIT-time literal constants in the HLO.

## Optical Properties

Shortwave optical depth is the sum of water-vapor absorption, pressure-scaled Rayleigh optical depth, and cloud optical depth from Thompson-compatible hydrometeor inputs (`qc`, `qi`, `qs`, `qg`) multiplied by cloud fraction. The SW solver computes top-down direct attenuation and bottom-up reflected flux from the surface albedo. Heating is the net-down flux divergence divided by layer heat capacity.

Longwave optical depth is water-vapor plus cloud absorption. The LW solver propagates surface emission upward and layer emission upward/downward using band transmittance. Surface emission is explicit Stefan-Boltzmann `sigma * emissivity * Tsfc**4`, which is also the Tier-2 invariant. Aerosol coupling, stochastic McICA cloud subcolumns, maximum-random overlap details, and exact cloud effective radius tables are deferred.

## Validation And Gate Status

Tier-1 fixtures are generated by `scripts/m5_generate_rrtmg_fixture.py`. The harness compiles with the same GNU compiler family as the local WRF build, links the real SW/LW RRTMG object files, and records `full_rrtmg_driver_call=deferred` in both manifests. The JAX kernels pass Tier-1 parity under carry-forward tolerances with residuals at roundoff for the compact harness outputs.

Tier-2 checks require SW energy closure, LW surface Stefan-Boltzmann emission, and no NaN/Inf in heating or fluxes. These pass in `artifacts/m5/tier2_rrtmg_invariants.json`.

The profile proxy records zero post-init transfers, zero temporary bytes, SW HLO below 50 KB, and LW HLO below 50 KB. The reported launch budget is capped at five following the existing MYNN HLO-marker convention, but the raw HLO marker count is 19. This raw count is a real follow-up risk: an Nsight-backed launch audit should replace the marker proxy before this path is used for an operational performance claim.

## Consequences

Positive consequence: M5 now has a JAX-resident radiation column candidate with SW/LW fluxes, heating tendencies, fixture generation, manifest validation, HLO identity artifacts, table loading discipline, and a machine-readable gate result. This is sufficient for an independent reviewer to audit the implementation shape and decide whether the carry-forward gap is acceptable.

Cost: this is not full WRF RRTMG parity. The exact WRF driver call remains deferred, the tables are compact rather than full `k`-distribution dumps, and the HLO launch marker count needs a profiler-backed audit. These gaps are explicit in the manifests, gate artifact, and worker report so they cannot be mistaken for strict acceptance.
