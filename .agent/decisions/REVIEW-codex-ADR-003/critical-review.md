# Critical Review — ADR-003 Dycore Precision

Decision: Accept with required fixes

## Decision

ADR-003 makes the right immediate call: retain fp64 for all M4 prognostic storage and dycore arithmetic, and do not authorize mixed precision in the current runtime. I accept that decision because it is conservative, consistent with ADR-002's fp64 default, and supported by the M4 fp64 proof objects. I do not accept the draft as final architecture text without fixes, because the future downcast plan currently outruns the evidence quality of M4 and does not give M5 workers a sharp enough proof gate.

## Top three structural concerns

1. The ADR treats M4 fp64 advection evidence as the launchpad for future downcasts without carrying forward the three documented M4 residual limits: acoustic is a reduced proxy, Tier-2 "mass" is a `theta_total` surrogate, and Tier-3 does not validate 2D velocity cross terms.

2. The downcast plan is procedural rather than executable. It says "dedicated precision sprint" and "fp32 rerun required", but does not name an artifact matrix, variable-specific tolerances, required baselines, or pass/fail gates for each proposed precision class.

3. The plan does not integrate the M5 stop/go reality. The dycore already trips the launch-count gate and lacks register/local-memory evidence, while M5 Thompson is the first real branchy physics test. A precision ADR must prevent downstream workers from using "fp32 candidate" language as a performance escape hatch before correctness and profiler evidence exist.

## Findings

1. **Major — Downcast plan is stronger than the residual M4 evidence supports.** ADR-003 proposes fp32 candidates for `theta`, passive tracers, non-acoustic advection tendencies, and M5 physics tendency arithmetic (`.agent/decisions/ADR-003-dycore-precision.md:21`, `.agent/decisions/ADR-003-dycore-precision.md:26`, `.agent/decisions/ADR-003-dycore-precision.md:28`, `.agent/decisions/ADR-003-dycore-precision.md:32`). But the M4 closeout explicitly limits the evidence: acoustic validation is proxy-only (`.agent/decisions/MILESTONE-M4-CLOSEOUT.md:60`), Tier-2 mass is `theta_total` rather than WRF-canonical `mu`/density mass (`.agent/decisions/MILESTONE-M4-CLOSEOUT.md:67`), and future closeouts must not carry it forward as mass-continuity proof (`.agent/decisions/MILESTONE-M4-CLOSEOUT.md:72`). Tester also flags that cross-term velocity advection has no manufactured-solution validation and Tier-3 sets `v=w=0` (`.agent/sprints/2026-05-19-m4-dycore-rk3-advection-acoustic/tester-report.md:211`). Required fix: narrow the downcast plan to "reduced-advection experimental candidates only" until an M4.x/M5 proof set includes true mass continuity, acoustic sound-wave validation for any pressure/acoustic-adjacent change, and a 2D cross-term convergence oracle.

2. **Major — The precision experiment gate is underspecified relative to policy.** `PRECISION_POLICY.md` requires mixed precision to be validated by variable and scheme, with tolerances documented per variable, fixture, and tier (`PRECISION_POLICY.md:5`, `PRECISION_POLICY.md:7`). ADR-003's downcast plan gives broad stages but no artifact names, no per-variable tolerance table at proposed precision, no required comparison against the fp64 reference, and no rule for whether a failed variable blocks only itself or the whole precision class (`.agent/decisions/ADR-003-dycore-precision.md:30`). Required fix: add an explicit matrix such as `field/component`, `candidate dtype`, `artifact paths`, `tier-1 tolerance`, `tier-2 invariant`, `tier-3 case`, `profile artifact`, and `authorization outcome`. Each row should default to "not authorized" until its artifact exists.

3. **Major — M5 physics downcast language is premature.** ADR-003 says physics tendency arithmetic is an fp32 candidate in M5 because column-local updates are "often less sensitive" (`.agent/decisions/ADR-003-dycore-precision.md:34`). ADR-005, however, deliberately selects Thompson as the first real JAX stop/go test and says the first real Thompson profile is the only readiness proof object (`.agent/decisions/ADR-005-first-physics-suite.md:42`, `.agent/decisions/ADR-005-first-physics-suite.md:44`). Thompson's frozen target includes hydrometeor species, number concentrations, water budget closure, positivity, latent-heating bounds, and fp64 tolerances (`.agent/decisions/ADR-005-first-physics-suite.md:21`, `.agent/decisions/ADR-005-first-physics-suite.md:27`, `.agent/decisions/ADR-005-first-physics-suite.md:28`). Required fix: state that M5 physics must first pass the frozen Thompson fp64 target; only then may a separate mixed-precision experiment run against the same Tier-1/Tier-2 fixture and profile gate.

4. **Major — The performance motivation for downcast is not tied to available profiler evidence.** The dycore profile reports 24 launches and null register/local-memory/occupancy data (`artifacts/m4/dycore_profile.json:17`, `artifacts/m4/dycore_profile.json:19`, `artifacts/m4/dycore_profile.json:21`, `artifacts/m4/dycore_profile.json:23`). The M5 dry-run trip is documented in closeout as `24 > 10`, with local memory and registers unknown (`.agent/decisions/MILESTONE-M4-CLOSEOUT.md:80`). Downcasting storage or arithmetic may reduce bandwidth, but it will not necessarily reduce launch count, and without register/local-memory evidence it may even hide the true bottleneck. Required fix: add a rule that no production downcast is justified by performance until a profile artifact reports launch count, register use, local memory, occupancy or stated profiler limitation, transfer audit, and an fp64-vs-candidate timing comparison on the same case.

5. **Minor — ADR-003 misquotes the fp64 evidence.** The draft says Tier-2 `mass_residual_relative = 0.0` in both rationale and validation evidence (`.agent/decisions/ADR-003-dycore-precision.md:12`, `.agent/decisions/ADR-003-dycore-precision.md:42`), but the artifact reports `1.937334197150901e-16` and identifies the mass field as `theta_total` (`artifacts/m4/tier2_invariants.json:6`, `artifacts/m4/tier2_invariants.json:7`). This is far below tolerance, but it is not zero and not canonical mass. The spacetime wall time is also slightly drifted: ADR-003 lists `631.020514992997` us (`.agent/decisions/ADR-003-dycore-precision.md:45`), while the committed artifact reports `634.0713601093739` us (`artifacts/m4/spacetime_budget.json:14`). Required fix: quote the JSON exactly or use rounded language such as "approximately".

6. **Minor — Status should reflect this review cycle.** ADR-003 currently says `Status: ACCEPTED` while critical review is pending and blockers would revert it (`.agent/decisions/ADR-003-dycore-precision.md:5`). This review is not a rejection, but it does require textual fixes before the ADR is clean. Required fix: revise status to "accepted with required fixes pending application" or apply fixes and then mark accepted with the review outcome recorded.

## Dissent

I do not dissent from fp64 retention for M4. I would have chosen an even narrower precision ADR: "fp64 remains the only authorized production dycore precision through M5-S1; fp32 is allowed only behind experiment flags with separate artifacts." I would not name M5 physics tendency fp32 as a candidate in ADR-003 before Thompson has passed once in fp64. If performance pressure forces earlier exploration, my preferred experiment is fp32 storage for passive tracer/theta advection with fp64 accumulation at conservation and coupling boundaries, compared against the fp64 M4 reference, but not merged into production until the M4 residual evidence gaps are closed.

## Closing recommendation

Accept ADR-003 only after tightening it into a conservative reference-precision ADR: correct the evidence values, explicitly scope all downcasts as unauthorised experiments, add a per-field artifact matrix, and bind M5 physics precision work to the ADR-005 Thompson fp64 gate first. The project should preserve fp64 as the production dycore default until the next precision sprint produces real fp32 proof objects and profiler data.
