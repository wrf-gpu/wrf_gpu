## Legend

**Character:**
* **ENDPOINT** — clean column endpoint: state→tendencies, vmap-able over columns,
  no new prognostic State, no special cross-column coupling. Plugs into the
  frozen `PhysicsTendency` contract with zero serial plumbing.
* **STATE-EXP** — state-expanding: adds prognostic species/carry that forces a
  serial `State`/IO/init/advection/nesting append patch (well-defined, but a
  manager-owned merge point, not parallelizable).
* **ARCH** — kernel/arch-forcing: data-dependent iteration, adaptive
  sub-stepping, huge lookup tables, or bin/spectral methods that stress
  fixed-shape JAX + fp32. These are the genuine research risks.

**Isolated-testable:** YES = single-step column savepoint parity (like our 12,
e.g. `tests/test_v060_pbl_ysu.py`); PARTIAL = column parity covers the kernel but
a coupling term (radiation feedback, sub-stepping cadence) only emerges in a run;
NO = correctness only visible in long full-coupling runs.

**Oracle:** CORPUS = free savepoints from the existing Canary corpus (only the
ported config: mp8/bl5/sfc5/lsm4/ra4/cu1); RERUN = needs a new pristine-WRF run
with a modified namelist (re-runnable today, gfortran serial build exists);
BUILD = needs a non-default WRF build option/external library before it can even
run.

**Tier:** A = easy endpoint, B = medium plumbing, C = hard arch.
**Score 1-10** anchored: 10 = F7 dycore rewrite; 7-8 = native-init (v0.3/v0.4);
5 = Noah-MP or RRTMG (DONE); 3-4 = a 2-moment MP; 2 = clean PBL/radiation
endpoint; 1 = Kessler.

## CRITICAL oracle fact (scopes everything)

**The entire Canary corpus runs ONE physics config**: `mp_physics=8 (Thompson)`,
`bl_pbl_physics=5 (MYNN)`, `sf_sfclay_physics=5 (MYNN-SL)`,
`sf_surface_physics=4 (Noah-MP)`, `ra_lw/sw=4 (RRTMG)`, `cu_physics=1 (KF, d01)`.
Verified in `/mnt/data/canairy_meteo/runs/wrf_l2/.../namelist.input`.

Consequence: the corpus gives FREE oracle savepoints for **exactly the 12 already
ported**. **Every other scheme needs a fresh pristine-WRF run** with a modified
namelist to extract its single-step column savepoints (via
`scripts/m6b0r_wrf_savepoint_extract.py` over wrfout slices). That CPU-WRF run is
the per-scheme oracle cost, and it competes for cores 4-31. Schemes flagged
BUILD additionally need a non-default `./configure` path or external data.

## 0.9.0 scope recommendation — the honest line

The user wants 0.9.0 to be a "mostly-full, mostly-tested WRF v4 port" and is
weighing **ALL-schemes vs common-only**.

**Recommendation: scope 0.9.0 to "all COMMON + all CLEAN-ENDPOINT" schemes, and
make ALL-schemes an explicit non-goal that becomes an open-ended trap.**

Where "all" becomes a trap, concretely:

1. **The arch-forcing tail is unbounded.** SBM, P3, NSSL, Jensen, NTU, CLM,
   CTSM, WRF-Chem are each a *milestone* (score 8-10) and several stress the
   fixed-shape-JAX + fp32 architecture itself. Six of them alone are >30 sprints
   and carry real "may not be portable at parity" risk. Chasing "all" means the
   long-tail exotics dominate the schedule while adding ~zero operational value
   for the Canary forecast mission.

2. **The oracle cost is real and serial.** Only the 12 ported schemes have free
   corpus savepoints. Every other scheme needs a fresh CPU-WRF run (cores 4-31,
   one at a time) to extract its oracle, and the BUILD-flagged ones (CAM,
   Goddard, FLG, CLM, CTSM, SBM-aerosol, Chem, aerosol-aware MP) need
   non-default builds + external data the project does not currently have. "All"
   silently pulls in a build/data-acquisition project, not just porting.

3. **Pairing constraints gate usefulness, not raw count.** A PBL is only
   forecast-usable with its mandated sfclay partner (MYJ↔MYJ-SL, GFS↔GFS-SL,
   QNSE↔QNSE-SL, TEMF↔TEMF-SL). Counting options overstates progress; count
   *usable pairs*.

### Concrete 0.9.0 target (defensible "mostly-full, mostly-tested")

Add to the 12 already done (~+18 schemes), all ENDPOINT or one-patch STATE-EXP,
all RERUN-oracle (no new builds), each single-step savepoint-gated:

* **MP:** Lin (2), WSM3 (3), WSM5 (4), Milbrandt2 (9), WSM7/WDM7 (24/26) — the
  common single/double-moment family, reusing ported WSM/WDM helpers.
* **PBL+sfclay pairs:** MYJ (2 + MYJ-SL 2), GFS (3 + GFS-SL 3), Shin-Hong (11),
  BouLac (8) — the common operational PBLs with their partners.
* **Cumulus:** G3 (5), SAS (4/14), BMJ (2, if its adjustment iteration ports
  cleanly) — the common convective schemes.
* **Radiation:** RRTM (1), Dudhia SW (1), Goddard SW (2), RRTMG-fast (24).
* **Land:** Slab (1), RUC (3), Pleim-Xiu (7) — completes the common LSM set.
* **Lake** (sf_lake): cheap, `lakemask` already a State leaf, real value for
  the Canary island domains.

That yields ~30 forecast-menu schemes — the WRF "common" menu essentially
complete and savepoint-tested — **without** opening the build/data/arch-risk
tail.

### Make these explicit, loud, documented 0.9.0 NON-GOALS (carry-over to v1.x+):

Spectral-bin MP, P3, NSSL, Jensen-ISHMAEL, NTU, all CAM-coupled schemes,
CLM4/CTSM/SSiB, FLG/Goddard-full/GFDL radiation, urban BEP/BEM, WRF-Chem, LES
3-D mixing. Each stays a `UnsupportedNamelistOption` loud-reject with a tracked
milestone ticket. Honest framing for the paper/README: **"v0.9.0 = the common
WRF-v4 physics menu, per-scheme savepoint-validated; the exotic/coupled tail is
explicitly out of scope and individually milestone-sized."** That is both
defensible and avoids the open-ended "all WRF schemes" trap.
