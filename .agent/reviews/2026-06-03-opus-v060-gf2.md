## cugd_* provenance (for the manager's v0.6.0 integration — NOT a blocker)

Pristine WRF `GFDRV` (the `cu_physics=3` call path) does NOT accept or update the
`cugd_*` arrays. In WRF those are wired through `G3DRV`/subsidence-spread, a
different driver. The v0.6.0 S0 adapter carries `cugd_*` as zero/diagnostic only;
**the correct integration carry from this scheme is the combined cumulus
tendencies** `RTHCUTEN/RQVCUTEN/RQCCUTEN/RQICUTEN` + `RAINCV`/`PRATEC` (added to
the model state) plus the shallow diagnostics `xmb_shallow/k22/kbcon/ktop_shallow`
and `ktop_deep`. The adapter already exposes these. Recommend the integrator drop
the `cugd_*` carry members for `cu_physics=3` (or document them as inert).

## Unresolved risks / notes

- **Single-column CPU reference, not vectorized/JAX-hot-path.** The GF closure is
  inherently sequential (level-by-level plume, downdraft iteration, 16-member
  dynamic control, beta-PDF `gamma()`), so the faithful artifact is sequential
  NumPy. The savepoint gate runs one column on CPU, so this is the correct
  equivalence object. A batched/JAX vectorization is a separate optimization
  sprint (GF is NOT load-bearing for v1.0.0 — KF done, Tiedtke covers the menu).
- **`imid_gf=0` path only** (matches GFDRV's compile-time `imid_gf=0`). The
  mid-level branch (`cup_gf` with `imid=1`, `get_inversion_layers` for mid tops)
  is not exercised by `cu_physics=3` and was ported only where shared
  (`get_inversion_layers` is used by the shallow path).
- **Oracle is a standalone WRF-module driver, not a coupled `wrf.exe` run**
  (`full_wrf_exe_run=false`), unchanged from the GPT lane — appropriate for a
  per-scheme column equivalence gate.
- Berry autoconversion (`autoconv=2`) and rain-evap (`irainevap=1`) branches are
  present in the source but inactive at the module defaults (`autoconv=1`,
  `irainevap=0`); the active branches are ported faithfully, the inactive ones
  are documented in-code but not all hand-translated (not reachable).
