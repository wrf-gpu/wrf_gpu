## Three non-obvious findings (all fixed / characterized)

1. **fp64 oracle build needed `-DDOUBLE_PRECISION` (REAL bug, FIXED).** The
   classic WDM6 `.F` selects `vsrec`/`vssqrt` (declared `real*4` in
   `frame/libmassv.F`) under the default `#ifndef DOUBLE_PRECISION`. Promoting
   the scheme's `real` to 8 bytes with `-fdefault-real-8` alone passes promoted
   real*8 `denfac` arrays to those real*4 args -> corrupted fall speeds (RAIN
   accumulators came out as denormals ~1e-317). Passing `-DDOUBLE_PRECISION`
   routes the macros to the real*8 `vrec`/`vsqrt` variants. After this the
   fp32<->fp64 oracle agrees to ~6e-6.

2. **Rain+Nr in-line sedimentation flux must use the STORED falk(k+1) (REAL
   bug, FIXED).** WDM6 does rain mass+number fallout in its own multi-substep
   loop (not the PLM). The interior flux-in term `dqr(k+1)` uses the falk
   computed when cell k+1 was processed (the previous loop step) and the
   already-updated qrs(k+1) -- NOT a recompute from the updated qrs(k+1). My
   first cut recomputed falk(k+1), which redistributed qr/nr in-column wrongly
   (case-1 qc/qr off ~3-6%). Carrying falk/falkn as state and using the stored
   value collapsed it to ~1e-6.

3. **Nrevp (NR->NCCN) ordering wipes snow if deferred (REAL bug, FIXED).** When
   rain fully evaporates (prevp == -qr/dtcld), WDM6 moves ALL rain number to
   CCN and zeroes Nr -- IMMEDIATELY inside the warm-rain do-k loop, BEFORE the
   cold-rain block. This is load-bearing: at cold subsaturated cells with rain,
   Nr->0 here, so the cold-rain rain-accretion terms (pracs/psacr/...) read
   nr=0 and do NOT fire. My first cut deferred the Nn transfer to after the
   conservation update, leaving nr>0 in the cold block -> pracs hit the qs
   clamp and wiped ~96% of snow in the melting-layer cases (2/4/5). Confirmed by
   instrumenting the Fortran oracle (DBG dumps of pracs/Nr at the cold-block
   entry). Moving the Nrevp nr->0 to immediately after prevp fixed cases 2/4/5
   to ~1e-6.

Note: the WRF source comment `qck1 ! 4706.08203` is STALE; the formula
`.104*9.8*peaut/denr**(1/3)/xmyu*den0**(4/3)` evaluates to 4534.67 (confirmed in
both fp32 and fp64). The formula is authoritative; the port uses 4534.67.

## Predeclared tolerances (frozen before comparison, never loosened)

t_abs=1e-2 K; q_rel=1e-2 / q_abs_floor=1e-7; n_rel=2e-2 / n_abs_floor=1e2;
precip_rel=1.5e-2 / precip_abs=5e-4 mm; re_rel=1e-2 / re_abs_floor=1e-7;
sr_abs=1e-2. The number tolerance is intentionally looser than mass (cube-root
lamda inversions + 1e1..1e10 dynamic range amplify fp32 roundoff); all fields
land far inside (number worst 4.1e-6, mass worst 8.1e-5).

## Nc/Nn State-leaf integration note (manager-owned, per S0 plan)

WDM6 needs the ADDITIVE State leaves `Nc` (qnc / QNCLOUD) and `Nn` (qnn /
QNCCN); `Nr` (qnr) already exists. Per the S0 plan this lane does NOT
materialize `State.__slots__` -- per-scheme parity is validated at the
SCHEME-FUNCTION level (`wdm6_run`: JAX-routine vs Fortran-routine on identical
inputs), which is done and PASSES.

`wdm6_physics_tendency` returns `state_replacements` for theta + the moist
species + `Nr` + `Nc` (both `Nc` and `Nr` are already valid
`STATE_TENDENCY_KEYS` via the registry `NUMBER_SPECIES`). `Nn` is also a valid
tendency key, but since the S0 plan designates `Nn` State materialization as the
manager-owned integration step, this adapter carries `Nn` in `diagnostics`
(no information lost) rather than presuming the leaf is wired. The manager's
integration patch must, in one reviewed change, append `Nc,Nn` to
`State.__slots__` / `STATE_FIELD_ORDER` / `PRECISION_MATRIX` / restart schema /
wrfout optional sourcing / nesting boundary allocation, then switch `Nn` from
`diagnostics` to `state_replacements` in `wdm6_physics_tendency`.

  multi-domain coupling + operational dispatcher wiring is the manager-owned
  serial integration step (S0 plan), not in this lane's scope.

  substeps; oracle columns use mstep=1. A GPU perf/transfer audit + a >120s-dt

  mixed-suite integration.

  diagflag/do_radar_ref, not in the PhysicsTendency contract -- not ported
  (matches WSM6 lane scope).

  dt=90s (one minor step). The multi-minor-loop path is exercised by the loop
  structure but not yet savepoint-gated at >120s.
