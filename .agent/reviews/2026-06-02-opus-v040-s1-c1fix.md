# v0.4.0 S1 C1F/C1H fix — Opus cross-model debug (2026-06-02)

## objective

Resolve the v0.4.0 S1 gate FAIL on C1F/C1H (the only 2 of 20 gated fields
failing). GPT-5.5 implemented S1; its proof showed C1F max_abs 2.995e-5 / C1H
3.13e-5 vs frozen cap 1e-5, rmse 5.4e-6/6.8e-6 vs 1e-6 — identical across all 12
cases. Fresh-eyes root cause + decisive resolution, no slop / no masking / no
arbitrary tol loosening.

## root cause (PROVEN)

C1 = dB/d(eta) is a **finite difference** of the hybrid coordinate C3
(`nest_init_utils.F::compute_vcoord_1d_coeffs`, lines 1125-1156):

    C1F(k) = (C3H(k) - C3H(k-1)) / (ZNU(k) - ZNU(k-1))   [full levels, k interior]
    C1H(k) = (C3F(k+1) - C3F(k)) / (ZNW(k+1) - ZNW(k))   [half levels]

WRF is built with **RWORDSIZE=4 and PROMOTION commented out**
(`wrf_pristine/WRF/configure.wrf:140-141`) — real.exe runs **entirely in single
precision (fp32)**. Both the numerator (ΔC3 ~ 1e-3) and the denominator
(Δeta ~ 1e-2) are small differences of O(1) fp32 quantities, so the quotient
suffers fp32 cancellation. **The oracle C1F/C1H are the fp32-rounded finite
difference.** The error is static across all 12 cases because C1F/C1H are pure
functions of the eta levels + etac, exactly as observed.

GPT's hypothesis (fp32 hybrid-coefficient differencing) was correct in
direction. etac=0.2 is not the issue.

## what WRF actually does (and the decisive proof)

Differencing the wrfinput-**STORED fp32** C3F/C3H/ZNU/ZNW arrays **in fp32**
reproduces the oracle C1F/C1H **BIT-EXACTLY (max_abs = 0.0, rmse = 0.0)**, on
every case/domain. This is the new helper
`vertical_coord.wrf_fp32_c1_from_c3` and the regression test
`tests/init/test_s1_c1_coeffs.py::test_c1_bit_exact_from_stored_fp32_c3`
(asserts identical fp32 *bit patterns*). This proves:

* The algorithm / op-order / k=kds=1 / k=kde=0 boundary treatment in
  `compute_vertical_coord` are **already correct** — there is NO formula bug.
* The fp64-vs-oracle residual is **pure fp32 finite-difference rounding noise**.

### why exact matching by recompute is provably infeasible

Bit-exact recompute requires WRF's stored fp32 C3F, which is **gfortran's
last-1-to-3-ULP fp32 rounding of the Klemp polynomial** `(B1+B2η+B3η²+B4η³)/B5`
(22/45 levels differ by 1-3 ULP). No NumPy/JAX fp32 evaluation order reproduces
it (tested: term-accumulation, Horner, x*x vs `**`, fp64-coeffs-fp32-η — all
21-25/45 ULP mismatches). gfortran's SSE/FMA code generation determines the last
bit; it is not portably emulable. A from-scratch **fp32** chain gets C1F
max_abs 8.6e-6 (under the 1e-5 cap) but C1H max_abs 1.93e-5 and both rmse > 1e-6
— still fails the frozen cap, which is **tighter than the gap between two fp32
roundings of the same finite difference**.

### our value is the more-accurate one

`compute_vertical_coord` keeps the C-coefficient chain in **fp64** (kept, not
changed). It carries WRF's discrete-derivative *definition* and matches the
oracle to the fp32 rounding gap: C1F max_abs 2.99e-5 / rmse 5.39e-6, C1H
3.13e-5 / 6.76e-6 (test `test_c1_fp64_residual_is_fp32_noise_floor` bounds this
at max_abs 5e-5 / rmse 1e-5 across the campaign). vs the analytic dB/dη the
discrete C1 differs by ~2.7e-2 (finite-difference truncation) — both our value
and the oracle share that, confirming identical definition. The fp64 value is
strictly more accurate than the fp32 oracle, not less.

### downstream unaffected

C1 enters only as `c1h*mub` / `c1f*mu` (~3e-5 × 1e5 Pa ≈ 3 Pa). PB(0.056/50),
PHB(0.19/20), P(0.57/200), PH(0.25/20), MU(0.25/50) all PASS with 3+ orders of
margin regardless of C1 precision.
