### Port notes (Dudhia SW)
- Faithful port of `module_ra_sw.F:SWPARA`. The K-loop is a sequential TOA->surface sweep (`SDOWN(K+1)` depends on `SDOWN(K)`, plus carried `TOTABS/OLDALB/OLDABC/WW/UV/DSCA/DABS/DSCLD`) implemented as a `lax.scan`.
- WRF flips arrays internally (`NK=kme-1-K+kms`); the port flips model-order columns to top-down for the sweep and flips the heating rate back, matching WRF exactly.
- Cloud albedo/absorption table interpolation (ALBTAB/ABSTAB 4x5 with XMUVAL nodes) reproduced bit-for-bit including the 1-based `IIL/IU/JJL/JU` fractional-index logic.
- `cssca = swrad_scat*1e-5` with namelist default `swrad_scat=1.0` (from `swinit`). Aerosols zero on the default path (`pm2_5_*` absent) so scattering reduces to the clear-sky `cssca` term — matches WRF default.
- WRF SWPARA does NOT use an ozone profile (that is an RRTM/RRTMG longwave/shortwave-G concept); ozone variation is therefore not a Dudhia-SW edge axis. Cloud + moisture + zenith are the active axes and are all exercised.

### Why RRTM LW JAX port is a scoped follow-on, not done here
Classic RRTM is a 16-band k-distribution radiative-transfer scheme (~7.6k Fortran LOC, embedded per-band DATA tables, external binary `RRTM_DATA` lookup asset). A faithful machine-precision JAX port is on the same order as the existing RRTMG-LW JAX port (2039 LOC + dedicated table-extraction tooling). It is not deliverable to the required fidelity in one sprint alongside a complete Dudhia SW port. The expensive, reusable groundwork (validated gold oracle + provenance + frozen tolerances + turnkey gate) is delivered so the follow-on is a pure-port sprint.

### Gotcha discovered building the RRTM oracle
`RRTM_DATA`/`RRTM_DATA_DBL` are **big-endian unformatted with 4-byte record markers** (WRF `configure.wrf BYTESWAPIO = -fconvert=big-endian -frecord-marker=4`). Without those gfortran flags the lookup-table read segfaults. fp32 build reads `RRTM_DATA`; fp64 build (`-fdefault-real-8`) reads `RRTM_DATA_DBL`. Serial DM stubs (`wrf_dm_on_monitor=.TRUE.`, no-op `wrf_dm_bcast_bytes` compiled with `-fallow-argument-mismatch`, bare-external `wrf_message`/`wrf_debug`) resolve link symbols without altering physics.

## fp32 vs fp64 / table handling
- **fp32 risk**: WRF compiles both radiation modules in REAL*4. The Dudhia long sequential per-layer accumulation carries single-precision dust; the JAX port runs fp64 so the fp32-vs-fp64 gap (~1e-6 rel on RTHRATEN, sub-mW/m^2 on GSW) is intrinsic to the precision difference, not a port error — confirmed by the fp64-vs-fp64 build matching to ~1e-15. Operational radiation is a held-rate theta tendency applied over the radt interval; a 1e-6 relative SW heating-rate difference is far below operational impact.
- **Table handling**: Dudhia's ALBTAB/ABSTAB are tiny embedded constants, ported verbatim as numpy arrays (no external asset). RRTM LW's k-distribution tables live in the external `RRTM_DATA` binary; a faithful JAX port must extract them (the existing `scripts/extract_rrtmg_tables.py` + `data/fixtures/rrtmg-tables-v1.npz` pattern is the template).
