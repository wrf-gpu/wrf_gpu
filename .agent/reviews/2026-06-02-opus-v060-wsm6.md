## Two non-obvious findings (both fixed / characterized)

1. **Post-sedimentation slope recompute (REAL bug, FIXED).** WRF calls
   `slope_wsm6` a second time on the POST-sedimentation qr/qs/qg before the
   snow/graupel melting block (mp_wsm6.F90 ~L661). My first cut reused the
   pre-sedimentation slopes for melting; this caused up to 6.5e-2 K T error and
   ~3% qs/qg error in the melting layer of the graupel-core case. Recomputing the
   slopes collapsed all prognostic errors to ~1e-6.

2. **Effective-radius floor flip = fp32 reference dust (NOT a port error).**
   `re_ice`/`re_snow` have categorical detection floors (background vs
   10.01/25 µm) that trip when a species exceeds ~1e-12 kg/kg. The canonical
   fp32 WRF scheme leaves single-precision round-off dust (e.g. qi=7.276e-12 =
   2^-37) at otherwise-empty cells, tripping the floor where the fp64 port (and
   an fp64 build of the SAME unmodified WRF scheme) correctly give 0. Against
   the fp64 oracle these cells agree to ~1e-9 rel. The report therefore gates
   the prognostic state + precip against the canonical fp32 oracle and the
   effective-radius DIAGNOSTICS against the fp64 oracle, and records the
   vs-fp32 error for each re field for full transparency. The PLM `find_kt`
   not-found fallback was also corrected (retain entry kt, matching WRF) which
   removed all real top-of-column residuals.
