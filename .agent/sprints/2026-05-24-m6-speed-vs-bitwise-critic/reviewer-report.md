# §1 — Is the two-mode separation sufficient?

Decision: mostly yes, but only if the separation is enforced as a recurring merge invariant, not as a single post-M6B6 cleanup sprint. The plan now says the savepoint harness is validation-only and that conflating it with operational mode caps speed by design (`PROJECT_PLAN.md:254`, `PROJECT_PLAN.md:256`). It also names the right operational escape hatches: mixed precision, strict-subset carry, fusion across savepoint boundaries, solver choice by profiler, GPU-optimal halo cadence, zero timestep-loop transfers, and diagnostic snapshots eliminated from production (`PROJECT_PLAN.md:260`, `PROJECT_PLAN.md:261`, `PROJECT_PLAN.md:262`, `PROJECT_PLAN.md:263`, `PROJECT_PLAN.md:264`, `PROJECT_PLAN.md:265`, `PROJECT_PLAN.md:266`). That is the correct invariant set. The weakness is timing and enforceability: the B-direct ladder explicitly tells M6B3 to adopt WRF scratch because savepoints demand it (`PROJECT_PLAN.md:226`), while the operational-mode ADR is not created until after M6B6 (`PROJECT_PLAN.md:277`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:3`). That creates a long period where validation-mode implementation shape can become the only shape everyone has tested.

Carry creep: §14.5.1 is explicit enough in intent but not tight enough in proof mechanics. It says operational carry is a strict subset and drops scratch not required by Tier-4 RMSE (`PROJECT_PLAN.md:261`), and it requires cited Tier-4 evidence before any field is added to operational carry (`PROJECT_PLAN.md:270`). The memory rule is even sharper: operational mode must drop scratch fields and every sprint merge must ask whether a deliverable bakes in CPU shape (`/home/enric/.claude/projects/-home-enric-src-wrf-gpu2/memory/feedback_gpu_optimized_core_primacy.md:15`, `/home/enric/.claude/projects/-home-enric-src-wrf-gpu2/memory/feedback_gpu_optimized_core_primacy.md:23`). The gap is that the M6-perf-design contract requires a table, but only after the full parity ladder has already taught the codebase to carry WRF scratch (`.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:49`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:62`). Tightening: every M6B1-B6 parity worker report should include a small "operational compatibility" table for new fields: validation-only, operational-required-with-evidence, or undecided. Undecided must not enter production state APIs.

Operator-boundary fusion lock: §14.5.1 names the risk correctly by permitting fusion across savepoint boundaries into XLA graphs or `lax.scan` bodies (`PROJECT_PLAN.md:262`), and §14.5 requires the whole timestep to be one compiled graph with RK/acoustic loops in `lax.scan`, with no `device_get`, host callbacks, or Python diagnostics in the operational timestep loop (`PROJECT_PLAN.md:249`). The memory instruction says to flag any deliverable requiring a savepoint boundary to remain a runtime fusion barrier (`/home/enric/.claude/projects/-home-enric-src-wrf-gpu2/memory/feedback_gpu_optimized_core_primacy.md:26`). The remaining gap is the phrase "may be fused"; "may" is too soft for a speed-preservation invariant. If an operator boundary is not fused, the project should require a profiler-based reason and an HLO/Nsight launch-count artifact. Otherwise a savepoint boundary can become a stable function boundary, pass correctness, and still degrade launch amortization. Tightening: replace "may be fused" with "must either fuse or carry a profiler-backed non-fusion exception in ADR-026."

Precision lock: the plan points to ADR-007 and says operational mode uses per-field fp32/bf16 where Tier-4 proves safe (`PROJECT_PLAN.md:260`). ADR-007 is strong where it has entries: mass, pressure, geopotential, acoustic accumulators, and pressure-gradient accumulation remain FP64 locked, while many winds, thermodynamic, tracer, and Thompson fields are FP32 candidates with operational RMSE gates (`.agent/decisions/ADR-007-precision-policy.md:30`, `.agent/decisions/ADR-007-precision-policy.md:36`, `.agent/decisions/ADR-007-precision-policy.md:61`, `.agent/decisions/ADR-007-precision-policy.md:84`). The precision policy also says production speed claims must use production precision (`PRECISION_POLICY.md:8`). The weakness is in the perf-design risk clause: if ADR-007 lacks a desired justification, the worker may write a small precision sub-document inside ADR-026 (`.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:124`). That is acceptable for planning, but not for production authorization. Tightening: ADR-026 may propose a downcast, but production operational code may only ship it if the relevant ADR-007 row or reviewed amendment exists and the Tier-4 evidence is attached.

Solver lock: §14.5.1 is explicit enough to avoid making serial Thomas sacred. It says validation mode uses WRF serial Thomas, while operational mode may use Thomas, PCR, or batched Thomas chosen by profiler plus Tier-4 envelope (`PROJECT_PLAN.md:263`). The perf-design contract also includes solver as a required per-operator column (`.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:53`). The risk is practical, not textual: M6B2 will implement and debug Thomas first (`PROJECT_PLAN.md:225`), and the recent coefficient work already warns that replacing the old runtime tridiagonal interface with WRF forward/back-sweep coefficients is a separate decision (`.agent/sprints/2026-05-24-m6b0r-defect-analysis-calc-coef-w/worker-report.md:47`, `.agent/sprints/2026-05-24-m6b0r-defect-analysis-calc-coef-w/worker-report.md:49`). Tightening: require ADR-026 to include either a PCR/batched-Thomas bakeoff proof or a profiler-backed reason that Thomas is not dominant for Canary 3 km. Tier-4 alone is not enough; solver changes also need conservation and boundedness checks because a solver can preserve T2/U10/V10 over 1h while injecting subtle vertical-mode noise.

State layout lock: §14.5.1 does not yet say enough. It protects carry fields and fusion boundaries, but not memory layout. The project constitution demands high-frequency state residency on GPU (`PROJECT_CONSTITUTION.md:9`, `PROJECT_CONSTITUTION.md:10`), and the plan warns that carry size is a performance problem even when correctness wants larger carry (`PROJECT_PLAN.md:250`). But savepoint serialization naturally wants named per-field arrays with metadata; ADR-025 freezes a savepoint schema with per-variable metadata and arrays under `/fields` (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-PROPOSED.md:34`, `.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-PROPOSED.md:35`). Nothing in §14.5.1 says operational layout must be chosen for coalescing, donation, aliasing, and XLA fusion rather than savepoint convenience. Tightening: add an explicit invariant that savepoint HDF5 layout is not the operational in-memory layout, and that ADR-026 must include a layout/memory-footprint proof for the operational state.

# §2 — Are the M6-perf-design acceptance gates the right ones?

The three gates are necessary but not sufficient. Tier-4 T2/U10/V10, wall-clock vs 28-rank CPU WRF, and zero H2D/D2H are exactly the right minimum gates because the user-level validation philosophy binds production fitness to operational RMSE rather than bitwise parity (`/home/enric/.claude/projects/-home-enric-src-wrf-gpu2/memory/feedback_validation_philosophy.md:17`, `/home/enric/.claude/projects/-home-enric-src-wrf-gpu2/memory/feedback_validation_philosophy.md:19`), and the constitution and performance targets forbid timestep-loop transfers (`PROJECT_CONSTITUTION.md:10`, `PERFORMANCE_TARGETS.md:5`). The perf-design contract mirrors that: operational mode must run the same Canary d02 1h, stay inside T2/U10/V10 envelope, beat CPU WRF by at least 1.2x, and commit an Nsight Systems trace (`.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:73`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:87`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:91`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:93`).

The 1.2x threshold is too low as a project performance target, but acceptable as a first perf-design kill switch only if it is labelled that way. The plan's original research synthesis rejected incremental OpenACC because a 5-7x ceiling is below the target of at least 8-10x (`PROJECT_PLAN.md:16`). ADR-001 records why the previous attempt failed: launch-bound architecture and register spilling collapsed the value proposition (`.agent/decisions/ADR-001-backend-selection.md:76`, `.agent/decisions/ADR-001-backend-selection.md:77`). A 1.2x pass would prove "not slower than CPU," not "GPU-native NWP value proposition." That is still useful between M6B6 and M6b, because correctness has just been recovered and the project needs an early architectural tripwire. But it must not become M7's success metric. Tightening: keep 1.2x for the first M6-perf-design sprint, but require ADR-026 to publish a measured path to the M7 target with launch count, memory traffic, precision, and hotspot plan. If the path is absent, fire the same step-back cadence even if 1.2x passes.

Tier-2 invariants should be gated, not merely observed. T2/U10/V10 can miss failure modes that become expensive later: mass drift, water-budget residuals, vertical-mode noise, or energy spectrum distortion. ADR-007's own failure shows that a speed ratio can pass while stability and Tier-2 invariants fail badly (`.agent/decisions/ADR-007-precision-policy.md:12`, `.agent/decisions/ADR-007-precision-policy.md:14`). The plan already treats validation as a four-tier pyramid, with physical invariants and timestep convergence below probabilistic consistency (`PROJECT_PLAN.md:18`). Therefore M6-perf-design should gate at least mass/pressure/water conservation residuals and finite/bounds audits on the operational variant. KE spectrum slope should be a diagnostic at M6-perf-design and a gate when solver/fusion changes materially affect momentum operators; otherwise it risks becoming a slow-to-debug M7 discovery.

Peak memory should be checked now. Carry creep and savepoint layout lock are memory problems before they are speed problems. The target hardware is single RTX 5090 first (`PERFORMANCE_TARGETS.md:8`), and operational mode is supposed to be the strict GPU-optimized production variant (`PROJECT_PLAN.md:256`). Waiting until M7 to discover that the strict subset does not leave 1 km headroom would allow a correctness-complete but commercially weak architecture to pass M6b. Tightening: add peak device memory, XLA temporary bytes where available, and 1 km projected headroom to Stage 4 or Stage 5. This is not a demand to run full 1 km in M6; it is a demand not to promote an operational design that obviously cannot fit.

# §3 — Is the perf-design sprint scoped correctly?

The per-operator table is the right default granularity, but it needs two extra views: per-RK-stage fusion groups and per-savepoint-boundary exceptions. The current contract names `calc_coef_w`, `advance_mu_t`, `advance_w` internal checkpoints, `advance_uv`, `calc_p_rho`, and `sumflux` (`.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:55`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:60`). That is a good level for carry, precision, and solver decisions because those choices are operator-specific and map to the savepoint ladder (`PROJECT_PLAN.md:223`, `PROJECT_PLAN.md:231`). It is also consistent with ADR-025's boundary registry, which already lists the relevant checkpoints from `calc_coef_w_pre/post` through acoustic and RK boundaries (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-PROPOSED.md:36`).

However, fusion is not naturally per-operator. A production timestep should care about compiled regions, loop nests, buffer donation, and launch amortization. The plan says the whole timestep should be one compiled graph and the acoustic/RK loops should be `lax.scan` rather than Python loops (`PROJECT_PLAN.md:249`), while the perf-design implementation requires a single `@jit` wrapping RK and acoustic scans (`.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:66`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:68`). Therefore ADR-026 should include a per-RK-stage fusion map: which validation operators collapse into each compiled region, what the HLO or trace says about launch count, and which boundaries remain materialized.

The table should also be finer at exceptions. Savepoint boundaries are acceptable in validation mode because ADR-025 uses them to recover operator truth (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-PROPOSED.md:13`, `.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-PROPOSED.md:14`). They are suspicious in operational mode. Any boundary that remains as a runtime synchronization, materialization, host callback, or separate launch needs a row explaining why it is numerically required or why the profiler says it is not important. This keeps the table from becoming too coarse to catch the precise failure mode the sprint exists to detect.

I would keep Stage 1 as per-operator, add a Stage 1b "compiled-region map," and add a Stage 1c "savepoint-boundary exception list." The worker should not be forced to make a row for every tiny boundary if it disappears in the HLO. But every boundary that survives operational lowering should be named and justified.

# §4 — Failure mode the project most likely won't catch in time

The most likely silent production failure is state layout lock with carry creep as its visible symptom. Operator-boundary fusion lock and H2D/D2H are more likely to be caught because the plan and contract demand `lax.scan`, no callbacks, zero transfers, and Nsight evidence (`PROJECT_PLAN.md:249`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:91`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:95`). Precision lock is also relatively visible because ADR-007 has an authorization matrix and production speed claims must use production precision (`.agent/decisions/ADR-007-precision-policy.md:57`, `PRECISION_POLICY.md:8`). Solver lock is risky, but a slow vertical solve should show up in the Stage 4 hotspot analysis if wall-clock misses (`.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:87`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:89`).

Layout and carry can pass those checks at small scale. A savepoint-shaped collection of named arrays can be resident on GPU, avoid explicit H2D/D2H, and still squander bandwidth, memory headroom, aliasing opportunities, and XLA fusion. It can also pass M6c's 24h Gen2 consistency because M6c is a statistical correctness gate, not a memory-traffic or 1 km feasibility gate (`PROJECT_PLAN.md:231`, `PROJECT_PLAN.md:316`). The recent M6B0-R work reinforces the danger: the harness has a correct HDF5 schema, per-variable metadata, and field arrays (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-PROPOSED.md:34`, `.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-PROPOSED.md:35`), while the direct running-WRF emission path is still incomplete and the savepoints came from real output plus a source-shaped reproduction rather than inside the running timestep loop (`.agent/sprints/2026-05-24-m6b0r-real-fortran-emission/worker-report.md:75`, `.agent/sprints/2026-05-24-m6b0r-real-fortran-emission/worker-report.md:78`). That is fine for validation progress, but it should not become the runtime state model by inertia.

The production symptom would be a model that passes parity, passes 1h RMSE, passes 24h consistency, and maybe even clears 1.2x on 3 km, but cannot leave enough memory and bandwidth headroom for 1 km daily operations. M7 then discovers that the "strict subset" is still too large or too fragmented. The project should catch this before M6b by making memory layout and peak-memory headroom explicit ADR-026 gates.

# §5 — Two-mode vs single-mode-with-feature-flag

The strongest argument for a single entry point with feature flags is bug-surface control. One function with `savepoint_emit=False`, `fp64_strict=False`, `kernel_fusion=True`, and solver flags makes it harder for validation and operational paths to drift semantically. It can also speed engineering because every operator fix immediately exercises both modes. Given the current defect history, this is not trivial: the `calc_coef_w` mismatch was a formulation/unit/staggering issue where the old path built meter-space coefficients while WRF used eta-coordinate pressure-mass factors (`.agent/sprints/2026-05-24-m6b0r-defect-analysis-calc-coef-w/worker-report.md:7`, `.agent/sprints/2026-05-24-m6b0r-defect-analysis-calc-coef-w/worker-report.md:11`). A single implementation core with mode flags might reduce the chance that the operational version quietly reintroduces a different formulation.

But the single-mode approach has a serious failure mode in JAX: flags become part of compilation surfaces, and validation affordances can leak into the production graph unless the code is very disciplined. The plan requires no Python diagnostics, no host callbacks, no `device_get`, and diagnostics only in a separate debug build (`PROJECT_PLAN.md:249`). The perf-design contract separately says no modifications to the savepoint harness or validation-mode build, no bitwise WRF parity commitment in operational mode, and no host/device transfers in the operational timestep loop (`.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:13`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:15`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:16`). These are easier to audit with distinct entry points.

I recommend the current two-mode-distinct approach, with shared pure numerical kernels underneath. Validation mode should own savepoint emission, strict fp64 parity, WRF scratch, and boundary materialization. Operational mode should own the production entry point, fused scans, carry subset, precision policy, and solver selection. Shared helpers are allowed where they are pure and do not impose layout, synchronization, or dtype. This is the best compromise between duplicate-code risk and performance-risk isolation. The current perf-design contract already points in this direction by creating `run_forecast_operational` and allowing operational-mode-only entry points without changing validation-mode semantics (`.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:26`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:27`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:28`, `.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:66`).

The key discipline is that "two-mode" must not mean "two unrelated models." The operational entry point should be traceable to validation-proven operator semantics, and any intentional deviation should cite Tier-4 plus Tier-2 evidence. Conversely, validation mode must never be allowed to define production layout or synchronization merely because it is easier to serialize and debug.

# §6 — Recommendation

Recommendation: `RATIFY-WITH-AMENDMENTS`.

Amendment 1: add a recurring operational-compatibility section to every M6B1-B6 parity sprint and closeout. It must classify new fields, boundaries, dtypes, and solver interfaces as validation-only, operational-approved-with-evidence, or undecided. Undecided items may not enter operational APIs. This implements the memory rule that parity sprint contracts must answer whether code runs in production and what the operational variant is (`/home/enric/.claude/projects/-home-enric-src-wrf-gpu2/memory/feedback_gpu_optimized_core_primacy.md:23`, `/home/enric/.claude/projects/-home-enric-src-wrf-gpu2/memory/feedback_gpu_optimized_core_primacy.md:24`).

Amendment 2: strengthen operator fusion from "may be fused" to "must fuse or carry a profiler-backed exception." ADR-026 must include a compiled-region map with HLO/Nsight launch evidence for RK and acoustic scans. This makes §14.5's whole-graph rule enforceable (`PROJECT_PLAN.md:249`, `PROJECT_PLAN.md:262`).

Amendment 3: add an explicit layout invariant: savepoint HDF5 schema and operational in-memory layout are separate contracts. ADR-026 must include operational state layout, peak device memory, temporary/aliasing evidence where available, and 1 km headroom projection. This closes the §14.5.1 state-layout gap.

Amendment 4: make precision authorization fail closed. ADR-026 may propose precision changes, but operational code cannot depend on a new downcast until ADR-007 or a reviewed amendment authorizes that field/path and cites Tier-4 evidence. This avoids a sprint-local precision sub-document becoming accidental production policy (`.agent/sprints/2026-05-25-m6-perf-design/sprint-contract.md:124`, `.agent/decisions/ADR-007-precision-policy.md:57`).

Amendment 5: add Tier-2 stability floors to M6-perf-design: finite/bounds, mass or dry-air continuity, water residual where physics is active, and conservation-sensitive checks for solver/fusion changes. T2/U10/V10 remain binding operational metrics, but speed cannot override invariant failure, as ADR-007's failed verdict already demonstrated (`.agent/decisions/ADR-007-precision-policy.md:12`, `.agent/decisions/ADR-007-precision-policy.md:14`).

Amendment 6: keep 1.2x as an M6 first-pass tripwire, but require ADR-026 to publish a measured path to the real M7 speed target, including hotspots, launch count, memory traffic, precision plan, and whether PCR/batched-Thomas is required. A 1.2x pass should not be advertised as satisfying the original GPU-native value proposition, because the plan rejected 5-7x ceilings as structurally insufficient against an 8-10x target (`PROJECT_PLAN.md:16`).

Dissent against this recommendation: the amendments may slow the B-direct ladder exactly when the project needs correctness momentum. One could argue that §14.5.2 already contains the decisive kill gate: if operational mode cannot beat CPU WRF within two perf-design sprints, the project reopens the savepoint-first framing (`PROJECT_PLAN.md:281`). From that view, extra B1-B6 operational compatibility paperwork risks distracting workers from the urgent job of recovering WRF small-step truth. I do not think that objection wins, because the requested additions are small proof hooks and ADR tables, not optimization sprints. They preserve the savepoint-first path while preventing validation architecture from hardening into production architecture before the perf sprint gets a chance to reject it.
