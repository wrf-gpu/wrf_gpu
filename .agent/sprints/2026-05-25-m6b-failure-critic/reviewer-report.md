## §3 - Does Hypothesis A cap max speed?

No. Carry expansion is a real cost, but on the stated RTX 5090 bandwidth budget it is marginal, not fatal. The sprint contract asks us to model the carry growth from about 14 fields to about 20 fields, or roughly +40%, on a Canairy d02 grid of `159 x 66 x 44 = 461 K cells` with RTX 5090 peak bandwidth near `1.8 TB/s` `.agent/sprints/2026-05-25-m6b-failure-critic/sprint-contract.md:56` `.agent/sprints/2026-05-25-m6b-failure-critic/sprint-contract.md:64`. The actual proof grid for two of the failed real runs is mass shape `[44, 66, 159]`, which is 461,736 cells `.agent/sprints/2026-05-25-m6b-honest-1h-canary/proof_bounds.json:59` `.agent/sprints/2026-05-25-m6b-honest-1h-canary/proof_bounds.json:65`.

The simple carry-footprint math is:

- Strict subset: `14 * 461,736 * 8 = 51,714,432 bytes`, or 51.7 MB decimal.
- Carry-expanded: `20 * 461,736 * 8 = 73,877,760 bytes`, or 73.9 MB decimal.
- Growth: `73.9 / 51.7 = 1.429`, so a purely bandwidth-bound implementation loses about 30% throughput, with expanded speed ceiling equal to `1 / 1.429 = 0.70` of the strict-subset ceiling.

If each carried field is read and written once per timestep, the lower-bound traffic is 103.4 MB per timestep for 14 fields and 147.8 MB per timestep for 20 fields. At 1.8 TB/s, that is about 0.057 ms per timestep for strict subset and 0.082 ms per timestep for carry-expanded. A 1 h run at 10 s timestep has 360 timesteps, so the lower-bound carry sweep budget is about 20.7 ms versus 29.6 ms for the whole hour. Even multiplying that by 10 full-state sweeps per timestep gives roughly 0.21 s versus 0.30 s. This is not the reason a GPU forecast would miss a 28-rank CPU WRF denominator near 687 s reported in ADR-026 `.agent/decisions/ADR-026-operational-mode-design-PROPOSED.md:34` `.agent/decisions/ADR-026-operational-mode-design-PROPOSED.md:40`.

The practical performance risks are launch count, XLA copies, FP64-heavy vertical solve, and transfer synchronization, not the six scratch families alone. The current first-step Nsight profile already shows 544 Device-to-Device copies and 59.023 MB D2D traffic in a failing 10 s reproduction `.agent/sprints/2026-05-25-m6b-honest-1h-canary/proof_nsys_transfers_inside_loop.json:8` `.agent/sprints/2026-05-25-m6b-honest-1h-canary/proof_nsys_transfers_inside_loop.json:18`. Those copies are in the same order as one carry sweep, so XLA aliasing and donation matter. But adding six resident scratch families is still a bounded memory-bandwidth penalty, not a structural defeat of the M7 8-10x value proposition. The project plan already anticipated this: carry size is a performance problem, not a correctness veto, and a correct large carry beats a sanitizer-dependent small carry `PROJECT_PLAN.md:243` `PROJECT_PLAN.md:250`.

Hypothesis B's bandwidth ceiling is better because it keeps the 14-field strict subset. In a world where the only issue is a localized RK or physics composition bug, the carry lower bound remains 51.7 MB footprint and 103.4 MB read+write traffic per timestep. That is worth preserving if bisection proves it. But the delta between 14 and 20 fields is so small relative to the target hardware that it should not block a correctness-restoring scratch promotion. The core rule is still GPU-optimized correctness, not minimalist state for its own sake. The principal directive says inefficient correct pilots are not enough, but also permits pilots that can be optimized `PROJECT_PLAN.md:245`. A fused, donated, carry-expanded operational state is optimizable; validation HDF5 savepoint emission is not, which is why I reject shipping validation mode as production.

## §4 - Recommended next sprint

Recommendation: **`SPLIT-INTO-TWO-SPRINTS`**. Do not combine carry expansion, bisection, and D2H lift in one implementation sprint. The theta-bounds failure is a correctness blocker, while D2H=53 is a constitutional transfer blocker. Both are serious, but they have different proof loops and different failure artifacts. Combining them would make it hard to tell whether a bounded first step came from carry expansion, substep-count correction, an accidental synchronization change, or a profiler-only harness change.

First sprint scope: **FIX-CARRY-EXPANSION with a bisection gate.** Add an operational carry object for `t_2ave`, `ww`, `muave`, `muts`, `ph_tend`, and `_save` families, initialized from the same real Gen2 state inputs used by the validation lane. Match the B-direct acoustic substep count for the bisection run before experimenting with a reduced operational count. Run one 10 s operational step on the three M6b real Gen2 run IDs, plus a single B6 golden-aligned savepoint comparison at post-stage or post-step boundary. Acceptance is finite/bounded theta and wind on all three runs, no regression of the B6 0.0 parity proof, and a written promotion table that changes only fields proven necessary by this M6b evidence. If the carry-expanded path still fails, stop and classify the root cause as Hypothesis B, with the first diverging boundary named.

Second sprint scope: **D2H lift.** Starting from the bounded operational path, rerun the warmed first-step Nsight profile with the raw report retained or parsed enough to identify call sites. Acceptance is zero H2D and zero D2H inside the captured operational timestep range, plus the same static grep used here. The second sprint should not change physical numerics except where a transfer lift requires replacing a host scalar/control path with a resident JAX value. After both pass, rerun M6b honest 1h on the three pinned runs and only then reopen Tier-4 RMSE, spatial divergence, and wall-clock gates.

I reject `PIVOT-OUT-OF-OPERATIONAL`. The steelman is that validation-mode parity is extremely strong: B6 proved coupled physics and boundary behavior with 0.0 max observed delta on three tiers `.agent/sprints/2026-05-25-m6b6-coupled-step-parity/worker-report.md:17` `.agent/sprints/2026-05-25-m6b6-coupled-step-parity/worker-report.md:23`. Shipping that path would prioritize correctness and preserve the hard-won savepoint ladder. The problem is that the project plan explicitly separates validation mode from operational mode and says validation HDF5/savepoint scaffolding may not become production carry or timestep synchronization `PROJECT_PLAN.md:254` `PROJECT_PLAN.md:266`. A production path that depends on validation scaffolding would violate the GPU-optimized-core directive before M7.

## §5 - Updated kill gates

The §14.5.2 kill gate should **stay as written and does not fire yet**. The current condition is operational mode failing to beat 28-rank CPU WRF within two perf-design sprints despite passing Tier-4 `PROJECT_PLAN.md:279` `PROJECT_PLAN.md:285`. M6b has not passed Tier-4. It did not even reach valid 1h RMSE or spatial divergence because all three real runs failed physical bounds after one 10 s step `.agent/sprints/2026-05-25-m6b-honest-1h-canary/worker-report.md:21` `.agent/sprints/2026-05-25-m6b-honest-1h-canary/worker-report.md:32`. The performance proof is explicitly `BLOCKED` because there is no valid 1h forecast wall-clock and transfer status is fail `.agent/sprints/2026-05-25-m6b-honest-1h-canary/proof_performance.json:1` `.agent/sprints/2026-05-25-m6b-honest-1h-canary/proof_performance.json:6`.

This is not evidence that savepoint-first was wrong. It is evidence that Amendment #1's operational promotion rule was necessary. The plan says operational mode drops scratch only when it is not required by Tier-4 RMSE, and that undecided fields may not enter operational APIs without follow-up evidence `PROJECT_PLAN.md:258` `PROJECT_PLAN.md:274`. M6b is exactly that follow-up evidence: the strict subset failed the first real Tier-2 physical-bounds gate. The correct response is not to weaken the kill gate or rewrite §14.5.2. It is to run the carry-expansion bisection and then re-test M6b honestly.

Dissent: keeping §14.5.2 unchanged may understate the management risk that ADR-026 overclaimed. ADR-026 recorded an operational 1h pass, zero transfers, and a huge speedup on a pinned case `.agent/decisions/ADR-026-operational-mode-design-PROPOSED.md:30` `.agent/decisions/ADR-026-operational-mode-design-PROPOSED.md:40`, but the first honest three-run real-state probe invalidated both physical bounds and transfer cleanliness `.agent/sprints/2026-05-25-m6b-honest-1h-canary/worker-report.md:7` `.agent/sprints/2026-05-25-m6b-honest-1h-canary/worker-report.md:12`. A stricter manager could amend §14.5.2 to count this as one failed perf-design attempt even before Tier-4. I do not recommend that because it would blur correctness failure and speed failure, but the next sprint should be treated as the final cheap chance to prove that operational mode is a controllable optimization of the savepoint truth rather than a separate unvalidated dycore.
