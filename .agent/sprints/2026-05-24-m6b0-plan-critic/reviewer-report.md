# M6B0 Plan Critic Reviewer Report

## §1 — Diagnosis confirmation

The consultant's diagnosis mostly holds: the strongest evidence says the current project cannot distinguish wrong recurrence, wrong staging, missing WRF scratch state, and wrong source-equation coupling at the level where the failure begins. The S2.1-redo real replay failed catastrophically at 1h, with T2/U10/V10 RMSE of 136.885 K, 106.419 m/s, and 102.232 m/s against Gen2 reference anchors, while the sanitizer audit recorded 17,237,505,011 candidate nonfinites and 2,811,641,892 clips (`.agent/sprints/2026-05-24-m6x-s2dot1redo-real-baseline/findings_real.md:20-24`, `.agent/sprints/2026-05-24-m6x-s2dot1redo-real-baseline/findings_real.md:34-56`). The same preserved replay explicitly lacks raw limiter arrays and per-substep RHS term arrays, so the existing evidence can say "bad by step 2" but not "which WRF small-step operator is first wrong" (`.agent/sprints/2026-05-24-m6x-s2dot1redo-real-baseline/findings_real.md:58-72`, `.agent/sprints/2026-05-24-m6x-s2dot1redo-real-baseline/findings_real.md:82-86`).

The S3 bug hunt also supports the diagnosis. Seven one-suspect A/B buckets all left the first nonfinite at step 2, including recurrence sign, mass-increment cap, metric-face change, acoustic-step sweep, physics disable, boundary disable, and pressure-branch verification (`.agent/sprints/2026-05-24-m6x-s3hunt-operator-bug-hunt/verdict.md:5-19`). That result does not prove the WRF small-step is intrinsically unportable; it proves the current tests are too coarse to isolate multi-term recurrence/staging faults. The source-mining table reinforces this reading because only one operator family is classified as "correct target shape," while five are approximate/blocked and two are missing accepted production evidence (`.agent/decisions/source_mining_operator_table.md:20-24`).

There is a counter-reading: the failure may be broader than instrumentation, because ADR-023 omitted WRF time-averaged scratch state by design and ADR-021's fuller WRF-shaped carry still failed when clamps were stripped (`.agent/decisions/blockers/M6-DYCORE-BLOCKER-MEMO.md:24-35`). But that counter-reading still points toward savepoint parity as the next experiment, not away from it, because both prior paths failed without a per-operator WRF oracle (`.agent/decisions/manager-reflections/PLAN-REFLECTION-2026-05-24-post-consultation.md:11-20`). I therefore confirm the diagnosis with one caveat: "missing instrumentation" is the immediate blocker; "WRF small-step parity may still be too expensive or structurally unsuitable" remains an open M6B1-B6 risk, not an M6B0 conclusion (`PROJECT_PLAN.md:259-264`).

## §2 — B-direct vs A-probe trade

Skipping A-probe is defensible under the approved B-direct plan. The post-consultation directive says A-probe is not scheduled because partial scratch could improve a surface metric while trapping the project in hidden staging work without attribution (`.agent/decisions/manager-reflections/PLAN-REFLECTION-2026-05-24-post-consultation.md:44-47`). PROJECT_PLAN §14 repeats that decision and gives A-probe only a future one-sprint kill-gated opening if new evidence appears (`PROJECT_PLAN.md:233-237`). Given that the bug hunt already ruled out seven local toggles and did not move the first nonfinite beyond step 2, another local implementation probe risks adding more unreviewed state before the oracle exists (`.agent/sprints/2026-05-24-m6x-s3hunt-operator-bug-hunt/verdict.md:7-21`).

The steelman for A-probe is still real. The blocker memo's manager addendum recommended a disposable A-as-probe first: add `t_2ave`/`ww`/`muave`, rerun S2.1-redo, continue only if T2 falls below 10 K, and pivot to B if it remains above 50 K (`.agent/decisions/blockers/M6-DYCORE-BLOCKER-MEMO.md:108-120`). The third-path scout also reported A as faster than B in estimated time-to-1h-coupled, because A targets a concentrated missing scratch family rather than all small-step recurrence state (`.agent/sprints/2026-05-24-m6x-third-path-substrate-scout/option_comparison.md:17-24`). If spare capacity existed and file ownership were isolated, a one-sprint A-probe could be a useful "cheap insurance" experiment, especially because WRF source lines show `MUAVE`, `MUTS`, `ww`, `t_2ave`, and `ph_tend` as central small-step quantities (`/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1102`, `/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1341`, `/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1451`).

I would not run A-probe now. The principal approved B-direct, the critic contract explicitly bars reopening the A/B/C/D choice, and M6B0 is an 8-16 hour harness sprint rather than a long implementation bet (`.agent/sprints/2026-05-24-m6b0-plan-critic/sprint-contract.md:19-26`, `.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:172-184`). A-probe should remain a pre-written fallback only if M6B0 cannot instrument WRF in two sprints or if B1/B2 proves the WRF oracle surface is unmanageably large (`PROJECT_PLAN.md:259-264`).

## §3 — M6B0 sprint contract review

The M6B0 contract is directionally sufficient to prove the harness, but it is too optimistic about the order in which trust is earned. Stage 1 builds an isolated instrumented WRF and protects the production `wrf.exe` by sha256 proof, which is the right first gate (`.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:56-70`). Stage 2 defines schema and I/O, and Stage 3 extracts real column and 16x16 patch bundles before Stage 4 proves the comparator can catch a deliberate perturbation (`.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:72-103`). That ordering front-loads WRF extraction before the comparator has demonstrated failure semantics. I would not move the real operator perturbation entirely before bundle extraction, because it needs realistic fields; I would add a Stage 2.5 synthetic savepoint dry-run that proves metadata validation, tolerance lookup, and comparator fail-closed behavior before Stage 3 spends time producing real bundles (`.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:72-79`, `.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:93-103`).

The Stage 1 boundary list is close but incomplete. It includes coefficient construction, `MU`/`MUTS`/`MUAVE`/`ww`, `t_2ave`, `ph_tend`, `advance_w`, pressure/geopotential restoration, one acoustic substep, and one RK stage (`.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:58-69`). It should explicitly include `advance_mu_t` entry/exit rather than only the scratch fields, because WRF updates `MU`, `MUDF`, `MUTS`, `MUAVE`, `ww`, and theta in that subroutine (`/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1102`, `/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1141`, `/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1175`). It should also split `advance_w` into internal checkpoints for RHS-ready, raw W update, tridiagonal forward sweep, backward substitution, Rayleigh damping, and final geopotential, because those are separate numerical phases in WRF (`/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1399`, `/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1477`, `/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1533`, `/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1546`, `/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1559`, `/home/enric/src/canairy_meteo/Gen2/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1581`). Divergence damping deserves at least a named savepoint or explicit "out of M6B0, in M6B6" deferral, because the source table identifies WRF pressure-memory `smdiv` and MPAS 3D divergence damping as a known concern (`.agent/decisions/source_mining_operator_table.md:14-16`). Lateral boundary application and microphysics coupling do not belong inside M6B0's minimum operator set, because PROJECT_PLAN puts physics/boundary-on coupled step parity at M6B6, not B0 (`PROJECT_PLAN.md:227-231`). The schema should nevertheless reserve phase labels for those later savepoints, because M6a deliverables include per-operator parity for a chain that eventually supports M6b coupled physics and boundary-on 1h (`MILESTONES.md:57-67`).

The Tier-1/Tier-2 storage progression is sane, but Tier 3 should not be "full d02 or defer" as the only next step. The contract requires one column and one 16x16 d02 patch, with full d02 10 acoustic substeps as stretch if storage stays under 20 GB (`.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:81-91`). ADR-025 already asks whether a golden small-domain should precede full d02 and defaults to yes (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-DRAFT.md:31-39`, `.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-DRAFT.md:61-67`). The milestone text also says run IDs must be pinned rather than wildcarding the d02 tree (`MILESTONES.md:75-77`). I recommend making a fixed "golden slice" mandatory after 16x16 and before full d02, with terrain, map-factor, and lateral-boundary metadata present even if boundaries are not yet applied. That slice is the bridge between a numerically clean patch and the real Canary geometry that triggered the blocker (`PROJECT_PLAN.md:203-210`).

The tolerance hooks are necessary but underspecified. Stage 4 requires a +1e-6 perturbation to be caught, and Stage 5 requires per-field max-abs delta versus chosen tolerances (`.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:93-111`). ADR-025 lists the tolerance ladder as unresolved and asks whether Tier-1 should be fp64 ULP-scale or relaxed for accumulations (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-DRAFT.md:31-39`, `.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-DRAFT.md:63-67`). M6B0 should emit a machine-readable tolerance ladder with units, dtype, abs/rel/ULP rules, accumulation exceptions, and a rule that deliberate perturbations are at least 10x the relevant pass tolerance for the field under test. Without that, a loose ladder can pass the negative test mechanically while hiding the exact errors the harness is supposed to expose (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-DRAFT.md:54-60`).

## §4 — ADR-025 draft review

ADR-025 has the right shape for a draft. It explicitly chooses B-direct with savepoint-first discipline, puts CPU WRF extraction before JAX comparison, and withholds 1h/24h reruns until 10-step coupled parity passes (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-DRAFT.md:9-18`). It also correctly says the exact instrumentation strategy, file format, schema field list, tolerance ladder, and golden-domain decision are unresolved during M6B0 (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-DRAFT.md:31-39`). That is appropriate because M6B0 is the evidence sprint, not a paper design exercise (`.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:113-122`).

I would not make ADR-025 commit to HDF5 versus NetCDF versus Serialbox before M6B0. The M6B0 contract already says HDF5 is default but allows NetCDF or an NPY bundle if the decision is recorded in ADR-025 (`.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:72-79`). The ADR should instead pre-commit to file-format capabilities: typed arrays, explicit endian/dtype preservation, schema version, chunking/compression notes, metadata validation, tamper detection, and a manifest that can list column, patch, and golden-slice savepoints. If Serialbox is unavailable, that should not block the harness; ADR-025's own open questions ask whether the Canairy WRF build has Serialbox or needs a hand-rolled path (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-DRAFT.md:61-67`).

Precision should be stricter for validation savepoints than for future operations. ADR-025 should pre-commit that WRF-emitted validation savepoints preserve fp64 or original WRF real-kind values with no downcast, because M6a's gate is sanitizer-off small-step parity and not operational speed (`MILESTONES.md:57-61`). It should not pre-commit that the eventual operational forecast is fp64-only, because PROJECT_PLAN §14.5 already distinguishes validation mode from operational mixed precision after savepoint and Tier-4 evidence (`PROJECT_PLAN.md:243-250`). The right rule is: savepoint truth is high precision and lossless enough for review; operational mixed precision is allowed only after per-field savepoint/Tier-4 proof and profiler evidence (`PROJECT_PLAN.md:266-272`).

## §5 — Schedule realism

The 10-17 sprint estimate is plausible only if M6B0 succeeds quickly and WRF small-step state turns out to be broad but finite. The plan currently sequences M6B0, B1-B6, M6b, M6c, and M6-perf as a long correctness-first ladder (`.agent/decisions/manager-reflections/PLAN-REFLECTION-2026-05-24-post-consultation.md:29-42`). Compared with the previous 5-9 sprint estimate, the plan correctly buys evidence instead of more sanitized forensics (`.agent/decisions/manager-reflections/PLAN-REFLECTION-2026-05-24-post-consultation.md:83-87`). My honest estimate is 14-22 sprints to M6 close, with a 25+ sprint tail if any of three risks materialize.

Risk 1 is WRF instrumentation friction. M6B0 touches a production-adjacent CPU WRF build, requires an isolated instrumented executable, and must keep the operational binary byte-identical (`.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:18-19`, `.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:56-70`). The plan has a two-sprint escalation gate for failing to patch WRF Fortran, which is necessary but could consume the first schedule overrun by itself (`PROJECT_PLAN.md:259-264`).

Risk 2 is savepoint surface explosion. ADR-025 already warns that full d02 savepoints across 10 steps, about 12 operators, and all 3D fields could be tens of GB (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-DRAFT.md:54-60`). The source table shows the likely surface includes mass update, vertical RHS, metric conversion, hyperdiffusion/smoothing, Rayleigh damping, divergence damping, time averaging, and tridiagonal solve families (`.agent/decisions/source_mining_operator_table.md:11-18`). If B1-B3 exposes more `_1`/`_save` interactions than expected, the project can spend many sprints adding state before it reaches M6B4 recurrence parity (`PROJECT_PLAN.md:223-229`).

Risk 3 is performance after correctness. The plan correctly defers optimization until after correctness, but it also requires no host/device transfers, whole timestep graph compilation, carry-size management, profiling before Triton/Pallas, and later mixed precision proof (`PROJECT_PLAN.md:243-250`, `PROJECT_PLAN.md:266-272`). A correct full WRF carry can still be slower than 28-rank CPU WRF, and PROJECT_PLAN makes that a re-open-performance gate at M6B5 (`PROJECT_PLAN.md:259-264`).

If 25+ sprints becomes the honest read, reduce scope explicitly instead of letting M6 sprawl. Keep M6a and M6b binding, but split M6c into a later pre-M7 release gate if 24h Gen2 consistency is the long pole; M6b already defines the sanitizer-free 1h Canary d02 proof with RMSE inside a pre-declared envelope (`MILESTONES.md:63-68`). Keep M6B0-B6 on one Canary d02 run ID and one golden slice until the savepoint ladder is stable, because the milestone file already warns not to wildcard the WRF backfill directory (`MILESTONES.md:75-77`).

## §6 — Option E (shadow GPU-WRF) priority

The case for running the E scout in parallel is strong because it is research-only, has no code edits, and does not promote E over B-direct (`.agent/sprints/2026-05-24-m6x-option-e-shadow-scout/sprint-contract.md:17-24`). Its deliverable is a license/cost/version/performance/maintenance comparison of AceCAST and FahrenheitResearch/wrf-gpu-port, plus a secondary-oracle assessment (`.agent/sprints/2026-05-24-m6x-option-e-shadow-scout/sprint-contract.md:47-80`). PROJECT_PLAN frames E as an insurance lane that does not replace B and may protect Canary continuity (`PROJECT_PLAN.md:239-241`). Because M7 stays on CPU WRF until M6b passes, an E scout can inform business-continuity planning without contaminating the savepoint harness (`.agent/decisions/manager-reflections/PLAN-REFLECTION-2026-05-24-post-consultation.md:79-81`).

The case for deferring E is also credible. The post-consultation reflection says E is optional and deferred unless the principal explicitly authorizes it (`.agent/decisions/manager-reflections/PLAN-REFLECTION-2026-05-24-post-consultation.md:52-55`). E-lane execution would create a second GPU-WRF architecture narrative while the approved B-direct path is trying to freeze WRF/JAX savepoint truth (`PROJECT_PLAN.md:211-231`). If E consumes manager attention, licensing cycles, or benchmark interpretation time before M6B0 has a schema, it can slow the decisive correctness work (`.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:172-184`).

My pick: allow the current E scout to run, but defer any E implementation/evaluation sprint until M6B0 produces either a working schema/comparator or a hard WRF-instrumentation blocker. The scout contract is bounded to 3-6 hours and public research only, so it is cheap information (`.agent/sprints/2026-05-24-m6x-option-e-shadow-scout/sprint-contract.md:82-96`). But E should not become a build or benchmark lane until B-direct has its first proof object, because the current plan's central claim is that CPU WRF savepoints are the oracle (`.agent/decisions/ADR-025-wrf-savepoint-bdirect-port-DRAFT.md:9-18`).

## §7 — Recommendation

Recommendation: `RATIFY-WITH-AMENDMENTS`.

1. Amend `.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:72-103` to add a Stage 2.5 synthetic-savepoint comparator dry-run before real bundle extraction. It should prove schema validation, tolerance lookup, and fail-closed perturbation behavior on a tiny synthetic savepoint, while preserving the real operator perturbation test after Tier-1/Tier-2 savepoints.

2. Amend `.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:58-69` to expand the minimum operator boundaries: explicit `advance_mu_t` entry/exit; `advance_uv` pressure-gradient/emdiv or an explicit deferral; `advance_w` internal checkpoints for RHS-ready, raw W update, forward sweep, backward substitution, Rayleigh damping, and PH final; and a named decision for WRF pressure-memory/divergence damping. Boundary and microphysics savepoints should remain M6B6 scope but be reserved in the schema.

3. Amend `.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:81-91` and `PROJECT_PLAN.md:223-231` to make a fixed golden small-domain slice mandatory after 16x16 and before any full d02 stretch. Include pinned run ID, terrain/map-factor metadata, and storage estimate; do not let full d02 be the first geometry beyond 16x16.

4. Amend `.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:97-111` to require `proof_comparator_tolerance_ladder.txt` or JSON to include per-field units, dtype, abs/rel/ULP thresholds, accumulation exceptions, and a rule that the deliberate perturbation magnitude exceeds the applicable pass tolerance by a fixed factor.

5. Amend `PROJECT_PLAN.md:259-264` to split the kill gates by phase: M6B0 failure is "cannot produce isolated WRF savepoints plus fail-closed comparator in <=2 sprints"; B1-B2 failure is "schema-valid savepoints expose too many early divergences for worker-only debugging"; M6B5 failure is the first appropriate point to re-open performance. The current "`>15 savepoints diverging at step 2 even when WRF carries are added`" gate is useful, but it should not be the M6B0 stop condition before WRF carries exist.

Dissent against this recommendation: a stricter critic could choose `RATIFY-POST-CONSULTATION-PLAN` with no amendments because the current M6B0 contract already has the major safeguards: isolated WRF build, column and 16x16 bundles, deliberate perturbation negative test, first operator parity proof, no-regression run, and no stabilizer/RMSE tuning (`.agent/sprints/2026-05-24-m6b0-wrf-savepoint-harness/sprint-contract.md:56-134`). Extra amendments may slow the worker by turning a harness bootstrap into a mini-framework design sprint, and the project already has a two-sprint WRF instrumentation escalation gate plus an external WRF expert review target (`PROJECT_PLAN.md:259-264`, `.agent/decisions/manager-reflections/PLAN-REFLECTION-2026-05-24-post-consultation.md:88-90`). I still prefer amendments because the cost is bounded and the failure mode they prevent is severe: a savepoint harness that produces files before it has proven it can reject wrong answers.
