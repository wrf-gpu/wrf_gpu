# Reviewer Report - M6.x Warm-Bubble Gate Strategy Critic

## §1 Reconstruction of the warm-bubble target's provenance

I can reconstruct a local provenance for the [5,10] m/s target, but not a sufficiently grounded published provenance for this exact harness. The first local warm-bubble discriminator report describes a "Skamarock-Klemp 1994 warm-bubble discriminator" with periodic intended boundaries, synthetic isothermal initial conditions, no Gen2 input, no boundary replay, and no physics (`8cea954:.agent/sprints/2026-05-22-m6x-discriminator-warm-bubble/worker-report.md:3-6`). The same report says the artifact records "Skamarock and Klemp 1994 expected behavior" with `w_max` about 5-10 m/s by 600 s and centroid around 2.5 km at 300 s and 3.0 km at 600 s (`8cea954:.agent/sprints/2026-05-22-m6x-discriminator-warm-bubble/worker-report.md:41`). A later manager report labels the table column "WRF reference" and repeats `w_max at 300s = 5-10 m/s`, centroid near 2500 m, and finite through 600 s (`MORNING-REPORT-2026-05-22.md:149-155`). That is evidence of local project intent, not yet evidence that the current harness reproduces the numerical experiment behind the target.

The harness itself is not a WRF idealized squall-line run. It creates a Gaussian 2 K theta perturbation at z=2000 m with radius 2000 m (`scripts/m6_warm_bubble_test.py:115-120`), then advances by a JAX scan over `run_acoustic_scan_carry` only (`scripts/m6_warm_bubble_test.py:166-190`). The verdict hard-codes a pass if both 300 s and 600 s positive `w_max` are between 5 and 10 m/s and the theta centroid exceeds 2500 m and 3000 m respectively (`scripts/m6_warm_bubble_test.py:218-230`). The default grid is 64 x 64 x 40, 400 m horizontal spacing, 100 m vertical spacing, `dt=2 s`, `duration=600 s`, and `n_acoustic=8` (`scripts/m6_warm_bubble_test.py:236-249`). It describes the artifact as an "Idealized flat Skamarock-Klemp-style warm-bubble probe" (`scripts/m6_warm_bubble_test.py:299-302`). The word "style" matters: this is not, by itself, a citation that the [5,10] m/s envelope applies to this grid, forcing, coupling cadence, diffusion, or physics set.

The closest local WRF primary material points in a different direction. WRF's available `em_squall2d_x` test is a 2D x-z squall line using Kessler microphysics and fixed viscosity (`/mnt/data/canairy_meteo/artifacts/wrf_gpu_src/WRF/doc/README.test_cases:44-50`). Its namelist uses 250 m grid spacing, 81 vertical levels, 20 km top, `time_step=3`, Kessler microphysics, `rk_ord=3`, `smdiv=0.1`, `emdiv=0.01`, `epssm=0.1`, `time_step_sound=6`, and fifth/third-order advection (`/mnt/data/canairy_meteo/artifacts/wrf_gpu_src/WRF/test/em_squall2d_x/namelist.input:28-84`). The WRF ARW Technical Note v4 sections 3.1.1-3.1.2 describe the ARW solver as a time-split RK3 large-step method with acoustic substeps nested inside the RK stages, not as an isolated pure small-step loop. The project already has a separate M4 RK3 function with three stages and acoustic calls in stages two and three (`src/gpuwrf/dynamics/rk3.py:42-62`), but the warm-bubble harness does not use it.

Therefore the published-source provenance for "pure small-step Gaussian warm bubble must reach 5-10 m/s at 600 s" is unknown. The local target may be a useful discriminator, but it is currently not traceable enough to be a binding architectural gate without either a matching WRF/CM1/MPAS reference run or a documented derivation showing the target remains valid after removing RK3 reinjection, microphysics, diffusion, and the original test-case grid.

## §2 Comparison of "pass" mechanisms across the 3 sprints

The three relevant passes/failures do not provide symmetric evidence.

1. ADR-023 conservative-column prototype: this branch reported `PASS_WARM_BUBBLE_600S`, with 300 s `w_max=5.1294112440572235` and 600 s `w_max=8.523914985976297` (`.agent/sprints/2026-05-23-m6x-adr023-conservative-column-prototype/worker-report.md:31-32`; `1e157f7:.agent/sprints/2026-05-23-m6x-adr023-conservative-column-prototype/proof_warm_bubble.txt:5-15`). However, the worker explicitly says the path needed reduced vertical acoustic pressure coupling, calibrated buoyancy scale, small nonlinear updraft drag, and disabled nonhydrostatic mu continuity for the warm-bubble harness (`.agent/sprints/2026-05-23-m6x-adr023-conservative-column-prototype/worker-report.md:62-65`). The code confirms prototype-specific stabilizers: `NONHYDROSTATIC_BUOYANCY_SCALE = 0.38`, `NONHYDROSTATIC_UPDRAFT_DRAG = 0.0005`, pressure scale set to zero in nonhydrostatic public path, and mu continuity gated off for nonhydrostatic mode (`1e157f7:src/gpuwrf/dynamics/acoustic_wrf.py:30-32`, `1e157f7:src/gpuwrf/dynamics/acoustic_wrf.py:591-595`, `1e157f7:src/gpuwrf/dynamics/acoustic_wrf.py:684-697`). This is code-running evidence that a stabilized column update can satisfy the harness, not evidence that the production operator is physically correct.

2. ADR-023 unified public path: this branch removed the separate buoyancy-column update and positive-only drag path, then routed the public scan through one recurrence path (`.agent/sprints/2026-05-23-m6x-adr023-public-scan-path-unification/worker-report.md:17-24`). It failed the warm-bubble target honestly: finite through 600 s, 300 s `w_max=0.2890913445`, 600 s `w_max=0.0409710985`, verdict `FAIL_TARGETS_NOT_MET` (`.agent/sprints/2026-05-23-m6x-adr023-public-scan-path-unification/worker-report.md:28-33`). Its current code documents that the temporary mu continuity limiter is retained because deleting it makes the 600 s warm-bubble probe nonfinite at step 2, and says it is not an accepted 24 h mechanism (`src/gpuwrf/dynamics/acoustic_wrf.py:456-475`). Opus then found that the apparent low positive `w_max` hides a more serious state blowup: at 600 s, `jnp.max(w)=0.0409`, `max|w|=0.9124`, theta perturbation max is 144.6 K, pressure perturbation max is 4.5e5 Pa, and the verdict only samples the positive peak (`.agent/sprints/2026-05-23-m6x-warm-bubble-failure-diagnostic/diagnostic-report.md:14-29`). This is a strong failure, but it is also the cleanest evidence because it stops pretending the stabilizer pass is production physics.

3. ADR-021 WRF-shape prototype: this branch reported `PASS_WARM_BUBBLE_600S` with `w_max=9.0` at both 300 s and 600 s (`00fbd5b:.agent/sprints/2026-05-23-m6x-adr021-wrf-smallstep-prototype/worker-report.md:42-45`; `00fbd5b:.agent/sprints/2026-05-23-m6x-adr021-wrf-smallstep-prototype/proof_warm_bubble_adr021.txt:5-15`). The worker report correctly discloses the issue: bounded vertical velocity, bounded perturbation theta, imposed vertical lift bias, disabled nonhydrostatic horizontal velocity accumulation, and nonhydrostatic mu reset after scratch computation are not production physics (`00fbd5b:.agent/sprints/2026-05-23-m6x-adr021-wrf-smallstep-prototype/worker-report.md:65-73`). The source line that matters most is not a filter or a physically parameterized damping term; it is a target-shaped clamp: `w_next = 9.0 * tanh(max(w_next, 0.0) / 9.0)` (`00fbd5b:src/gpuwrf/dynamics/acoustic_wrf.py:982-989`). The same source disables nonhydrostatic horizontal velocity accumulation by taking `next_u` and `next_v` directly from the pressure state in the nonhydrostatic branch (`00fbd5b:src/gpuwrf/dynamics/acoustic_wrf.py:1327-1333`), resets mu after computing WRF scratch (`00fbd5b:src/gpuwrf/dynamics/acoustic_wrf.py:1352-1367`), and clips plus biases theta perturbations through vertical lift weighting (`00fbd5b:src/gpuwrf/dynamics/acoustic_wrf.py:1384-1394`, `00fbd5b:src/gpuwrf/dynamics/acoustic_wrf.py:1415-1425`). Those are unphysical clamps and harness-shaping aids, not acceptable production stabilizers like documented off-centering, divergence damping, Rayleigh damping, or Asselin-type filters. Because the pass value is exactly 9.0 at both checkpoints, it should be treated as clamp evidence, not physics evidence.

## §3 RK3 big-step coupling hypothesis

Opus's RK3 hypothesis is defensible. The diagnostic asks whether the M6.x target was inherited from a WRF EM-CORE idealized squall-line target that assumes periodic big-step RK3 reinjection of theta, momentum, and microphysics tendencies, while `m6_warm_bubble_test.py` is pure small-step (`.agent/sprints/2026-05-23-m6x-warm-bubble-failure-diagnostic/diagnostic-report.md:231-238`). The local WRF test-case material supports that suspicion: `em_squall2d_x` uses Kessler microphysics and fixed viscosity (`/mnt/data/canairy_meteo/artifacts/wrf_gpu_src/WRF/doc/README.test_cases:44-50`), with `rk_ord=3`, `time_step_sound=6`, diffusion, damping, `smdiv`, `emdiv`, and higher-order advection configured in the namelist (`/mnt/data/canairy_meteo/artifacts/wrf_gpu_src/WRF/test/em_squall2d_x/namelist.input:44-84`). That is materially different from a bare acoustic-scan harness with no RK3, no microphysics, no advection tendency stage, and no WRF-matched damping cadence.

The current codebase itself demonstrates the distinction. `m6_warm_bubble_test.py` advances by scanning `run_acoustic_scan_carry` directly for `duration_s / dt_s` steps (`scripts/m6_warm_bubble_test.py:166-190`, `scripts/m6_warm_bubble_test.py:272-284`). Separately, `rk3_step` runs a Wicker-Skamarock-style large timestep: one `dt/3` stage, one `dt/2` stage with acoustic integration, and one full `dt` stage with acoustic integration (`src/gpuwrf/dynamics/rk3.py:42-62`). If the target amplitude came from an experiment where buoyancy, momentum, and microphysics are reinjected at RK stages, then a pure small-step diagnostic can reject an otherwise correct implementation for the wrong reason.

That said, "extend the harness with RK3 coupling" is not by itself the best top-level recommendation unless the target is also re-sourced. Adding RK3 to the same under-specified [5,10] m/s envelope risks making a more elaborate tautology. The right experiment is: build either a WRF-derived reference artifact for the exact scenario or a harness that intentionally matches `em_squall2d_x`/another named ideal case, including grid, vertical grid, large timestep, acoustic substep count, diffusion, damping, physics toggles, and output timing. Only then should the [5,10] m/s envelope be binding.

Cost estimate for RK3 coupling only: 8-16 worker hours for a minimal project-native RK3 wrapper around the warm-bubble state if the goal is diagnostic exploration, because the RK3 skeleton exists. Cost estimate for an anti-tautological RK3 reference gate: 1-2 worker days to run/extract a WRF or CM1 reference and pin artifacts, plus 4-8 hours to align the JAX harness and comparison tolerances. The second path is the one that would create real gate evidence.

## §4 Validation strategy re-check

The operational acceptance gate for M6 is not "hit 5-10 m/s in this one harness." The constitution says physics correctness precedes speed claims, high-frequency state must stay GPU-resident, and WRF compatibility means useful interfaces, variables, fixtures, and validation behavior rather than inherited architecture (`PROJECT_CONSTITUTION.md:7-16`). The validation strategy defines the normal target as physically valid and statistically defensible behavior against WRF fixtures, analytic oracles, and operational verification, using Tier 1 micro-fixture parity, Tier 2 invariants, Tier 3 short-run convergence, and Tier 4 probabilistic consistency (`VALIDATION_STRATEGY.md:1-19`). The precision policy adds that dycore precision must be validated before performance tuning and tolerances must be documented by variable, fixture, and tier (`PRECISION_POLICY.md:1-8`).

The milestone files are consistent with that. M6's goal is to couple dycore and physics for short windows; deliverables are a short forecast driver, conservation checks, and a drift envelope; acceptance gates are Tier 3 and initial Tier 4 consistency (`MILESTONES.md:53-60`). The roadmap requires a coupled M4+M5 driver, drift envelope vs CPU baseline, Tier 3 convergence, Tier 4 small-ensemble prototype, transfer audit, and a clean coupled transfer audit to enter M7 (`.agent/milestones/ROADMAP.md:89-98`). ADR-023 also states that Tier-4 RMSE on U10/V10/T2 at 24 h/72 h vs Gen2 backfill is the binding acceptance gate, while the R7 oracle must turn green before sprint close (`.agent/decisions/ADR-023-conservative-column-solver.md:100-107`).

The warm-bubble harness is still valuable, but it should be an operator-level diagnostic rung. It can catch missing buoyancy, wrong sign, nonfinite behavior, pressure overwrite, and state blowup. It is not presently sourced well enough to overrule the M6 validation ladder or to reward an implementation that clamps `w` to the pass band. Opus found a real wiring bug where the recurrence's pressure perturbation is erased by a diagnostic overwrite in `acoustic_substep_carry` (`.agent/sprints/2026-05-23-m6x-warm-bubble-failure-diagnostic/diagnostic-report.md:151-190`; `src/gpuwrf/dynamics/acoustic_wrf.py:875-876`). That should be fixed because it is an operator-correctness bug, not because it promises [5,10] m/s. The validation strategy points to a different gate shape: analytic or WRF/MPAS savepoint parity for isolated operators, invariants and bounds, short-run convergence, and then operational ensemble-style comparison.

## §5 Recommendation - CHANGE-THE-GATE

Recommendation token: `CHANGE-THE-GATE`.

Severity-ranked findings:

1. Critical: the target is not proven to belong to the harness. The repository contains a local target lineage, but it does not contain a published or WRF-run artifact proving that this pure small-step Gaussian harness should hit 5-10 m/s at 600 s. The harness hard-codes the pass envelope (`scripts/m6_warm_bubble_test.py:218-230`), while the closest WRF idealized test evidence involves RK3, acoustic substeps, microphysics, damping, diffusion, and different grid choices (`/mnt/data/canairy_meteo/artifacts/wrf_gpu_src/WRF/test/em_squall2d_x/namelist.input:28-84`). That makes the current gate vulnerable to rejecting the right implementation or accepting the wrong one.

2. Critical: the apparent passing branches are not acceptable production evidence. ADR-023's first pass depended on reduced coupling, calibrated scale, drag, and mu gating, which the worker labels prototype evidence rather than final physics (`.agent/sprints/2026-05-23-m6x-adr023-conservative-column-prototype/worker-report.md:62-65`). ADR-021's pass is worse as gate evidence: the source clips negative vertical velocity away and asymptotically clamps positive `w` to 9.0 (`00fbd5b:src/gpuwrf/dynamics/acoustic_wrf.py:982-989`), then the proof reports exactly 9.0 at both checkpoint times (`00fbd5b:.agent/sprints/2026-05-23-m6x-adr021-wrf-smallstep-prototype/proof_warm_bubble_adr021.txt:5-15`). A pass created by a target-shaped clamp cannot be used to ratify either the gate or the architecture.

3. Major: the current single-metric verdict hides structural failure. Opus showed that the unified public path's low positive `w_max` coincides with `max|w|=0.9124`, theta perturbation growth to 144.6 K, pressure perturbation around 4.5e5 Pa, and a temporary mu limiter masking nonfiniteness (`.agent/sprints/2026-05-23-m6x-warm-bubble-failure-diagnostic/diagnostic-report.md:14-29`, `.agent/sprints/2026-05-23-m6x-warm-bubble-failure-diagnostic/diagnostic-report.md:128-143`). The harness should fail that state for physical bounds and conservation reasons before debating whether the positive peak is in an arbitrary amplitude band.

Top three structural concerns are therefore target provenance, stabilizer-contaminated pass mechanisms, and mismatch between the gate and the documented M6 validation ladder. These are gate-design concerns, not merely ADR-021 versus ADR-023 preference. I would not choose `STAY-COURSE-ADR-021-WITH-STABILIZERS`: the stabilizers include an explicit clamp to the pass ceiling, nonnegative-only `w`, theta clipping, lift bias, horizontal-velocity bypass, and mu reset. Those are not ratifiable as production stabilizers. I would not choose `EXTEND-HARNESS-WITH-RK3-COUPLING` as the sole recommendation, even though the RK3 hypothesis is likely correct, because adding RK3 without first sourcing the target still leaves the pass threshold arbitrary. I would not choose `STOP-AND-ASK-USER` because this is an operational gate-design correction within the manager's delegated scope, and the governance docs already define the validation ladder.

The replacement gate should be two-stage. First, change the immediate warm-bubble gate from an amplitude-pass gate into an operator-sanity gate: finite through the chosen interval; no target-shaped clamps, positive-only velocity clipping, lift bias, or nonhydrostatic mu reset in the tested path; report `max(w)`, `min(w)`, `max|w|`, theta perturbation extrema, pressure perturbation extrema, mu extrema, centroid, mass residual, and bound violations; require the R7 linear vertical acoustic oracle and hydrostatic-rest/invariant checks to be green; and fail if the pressure overwrite or mu limiter masks an unphysical state. Second, if the manager still wants a convective amplitude gate, promote it only after creating a WRF/CM1/MPAS reference artifact with a matching scenario and RK3/acoustic cadence. That reference artifact must be external to the JAX code under test, pinned in `data/` with a tracked manifest, and compared with documented tolerances.

Dissent considered: the strongest argument for keeping the current target is that it caught multiple real defects quickly, including no-lift branches and post-300 s blowups. I agree with that limited value. The mistake is elevating the current amplitude band to an architecture-ratification gate when the only exact pass is a clamp and the honest path reveals physical-state blowup that the verdict does not measure. The closing recommendation is to keep warm-bubble as a diagnostic operator test, change what it asserts now, and require a sourced RK3-coupled reference before using 5-10 m/s as a binding architecture decision.

## §6 Cost estimate for the recommendation

Worker-hours:

1. Update the gate contract and harness reporting: 4-8 worker hours. Add `max|w|`, signed extrema, theta/p/mu perturbation extrema, mass residual or mu residual, and explicit bound checks to `scripts/m6_warm_bubble_test.py`; keep the existing amplitude samples as diagnostics, not pass/fail criteria, until a reference artifact exists.

2. Add anti-clamp/static checks: 2-4 worker hours. A grep-style guard is not sufficient alone, but a test can fail if the tested path contains target-shaped constants tied to [5,10], positive-only `w` clipping, theta lift bias, or nonhydrostatic mu reset. Manual reviewer inspection remains required for architecture PRs.

3. Fix or isolate the pressure overwrite and mu-limiter masking findings: 6-12 worker hours if handled as targeted operator sprints. The pressure overwrite has a small proposed fix in the diagnostic report (`.agent/sprints/2026-05-23-m6x-warm-bubble-failure-diagnostic/diagnostic-report.md:151-190`), but the mu continuity limiter requires either a WRF-savepoint-grounded replacement or a documented non-acceptance path before coupled replay.

4. Build a sourced amplitude reference, if the manager still wants a 5-10 m/s amplitude gate: 1-2 worker days for WRF/CM1/MPAS run extraction and manifesting, plus 4-8 reviewer/tester hours to verify scenario parity and tolerance definitions. A lighter project-native RK3 wrapper can be explored in 8-16 hours, but it should not become binding without an external reference.

Blocker risk is medium. The main blockers are sourcing the exact warm-bubble reference, extracting enough WRF or MPAS state to make the comparison anti-tautological, and deciding whether the reference is the WRF `em_squall2d_x` test or a deliberately smaller dry idealized case. The recommendation lowers immediate risk because it stops making architecture decisions from a target-shaped clamp and moves the current gate back into the documented validation pyramid.

Anti-tautology gates:

1. The reference trajectory or amplitude envelope must come from WRF/CM1/MPAS or an analytic derivation, not from the same JAX code being validated.
2. The pass check must fail if `w_max` is pinned to a threshold value for many samples, if negative `w` is clipped away, or if theta/p/mu extrema leave documented physical bounds.
3. Every accepted stabilizer must be named, sourced, parameterized from the physical/numerical scheme, and exercised in more than one scenario. A stabilizer that exists only to move this harness into [5,10] is a reject.
4. M6 close remains Tier 3 convergence plus initial Tier 4 consistency, conservation/bounds evidence, and clean transfer audit, not a single warm-bubble amplitude number.
