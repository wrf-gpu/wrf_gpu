# Reviewer Report - M6.x ADR-023 Three-Way Critic

Date: 2026-05-23
Role: critical-review
Branch: `critic/codex/m6x-adr023-three-way-critic`

## §1 Steelman of ADR-023

ADR-023 is the strongest of the three proposals because it keeps the c2 architecture boundary small while correcting the main physical defect in ADR-022. The project constitution explicitly says WRF compatibility means useful interfaces, variables, fixtures, and validation behavior, not inherited architecture, and it also says this is not a line-by-line WRF Fortran port (`PROJECT_CONSTITUTION.md:7`, `PROJECT_CONSTITUTION.md:9`). ADR-020 similarly chose a JAX dycore skeleton with named state, base-state, boundary, and metrics contracts rather than a direct WRF object graph (`.agent/decisions/ADR-020-c2-dycore-architecture.md:8-10`). Within that architecture, preserving the six-leaf `AcousticScanCarry` shape used by the current scan (`src/gpuwrf/dynamics/acoustic_wrf.py:43-76`) is a real design win if the vertical operator can be made physically credible.

The best case for ADR-023 is that the project does not need WRF's small-step scratch families as public scan contract, but it does need a conservative column solve. The c2-A2/A2.x review showed that the current vertical path is not a WRF `advance_w` port: it misses `ph_tend`, `ww`, `t_2ave`, `muave`, off-centering, and the WRF final `ph` weighting (`.agent/sprints/2026-05-22-c2-A2-A2x-bundle-review/reviewer-report.md:29-46`). It also showed that the current vertical tests were qualitative and passed implementations with wrong dynamics (`.agent/sprints/2026-05-22-c2-A2-A2x-bundle-review/reviewer-report.md:124-137`). ADR-022 fixed the carry problem but left a simplified `ph_new = ph_old + dt*g*w_new` update and asserted the omitted mass/geopotential weighting would not matter (`.agent/decisions/ADR-022-hybrid-vertical-operator-DRAFT.md:56-65`). ADR-023 correctly removes that weak simplification and instead proposes a coupled `(w, mu, theta, phi/exner)` column solve (`.agent/decisions/ADR-023-conservative-column-solver-DRAFT.md:10-18`).

On external precedent, ADR-023 has a reasonable pattern match. MPAS computes vertically implicit gravity-wave/acoustic coefficients before the RK/acoustic loop (`/mnt/data/canairy_meteo/artifacts/wsm6_gpu_port/MPAS_wsm6_GPU_for_CAG_clean/MPAS-Model-5.3/src/core_atmosphere/dynamics/mpas_atm_time_integration.F:437-475`) and explicitly says its acoustic step follows Klemp et al. 2007 using forward-backward vertically implicit integration (`/mnt/data/canairy_meteo/artifacts/wsm6_gpu_port/MPAS_wsm6_GPU_for_CAG_clean/MPAS-Model-5.3/src/core_atmosphere/dynamics/mpas_atm_time_integration.F:1824-1832`). The MPAS coefficient builder uses the off-centered time factor `dtseps = .5*dts*(1.+epssm)` (`.../mpas_atm_time_integration.F:1589-1591`), while the acoustic step uses `resm = (1.0 - epssm) / (1.0 + epssm)` (`.../mpas_atm_time_integration.F:2038-2041`), solves upward and downward through a tridiagonal recurrence (`.../mpas_atm_time_integration.F:2172-2182`), and then back-substitutes density and theta perturbations from the updated vertical momentum (`.../mpas_atm_time_integration.F:2201-2208`). That is good evidence for ADR-023's conservative column-solver family.

However, I do not accept ADR-023's current statement that MPAS Klemp 2007 is "mathematically equivalent" to the proposed Crank-Nicolson column solver with `epssm=0.1`. The equivalence is family-level, not yet equation-level. MPAS solves perturbation variables `ru_p`, `rw_p`, `rtheta_pp`, and `rho_pp` in its native vertical coordinate (`.../mpas_atm_time_integration.F:1828-1833`, `.../mpas_atm_time_integration.F:1889-1906`), while ADR-023 specifies WRF-hybrid denominators and a proposed `phi/exner` coupling in prose (`.agent/decisions/ADR-023-conservative-column-solver-DRAFT.md:40-47`). WRF's own `calc_coef_w` denominators are level-specific (`/mnt/data/canairy_meteo/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:624-646`), and WRF's `advance_w` couples phi RHS, off-centering, buoyancy, and final geopotential update through `mut/muts`, map factors, and `t_2ave` (`.../module_small_step_em.F:1341-1396`, `.../module_small_step_em.F:1477-1489`, `.../module_small_step_em.F:1581-1585`). ADR-023 can use MPAS as precedent, but it still needs a written discrete derivation mapping its variables to the JAX operator before claiming equivalence or a 5 percent savepoint target.

On Newton: the best defense of ADR-023's no-Newton v0 is that MPAS and WRF use linearized tridiagonal acoustic solves in this family, and the local immediate gate is the red linear acoustic oracle, not a fully nonlinear exner/EOS solve. The new oracle derives constant-coefficient linear equations and a dispersion relation (`src/gpuwrf/validation/analytic_oracles/vertical_linear_acoustic.py:3-25`) and checks phase/amplitude in tests (`tests/test_m6x_vertical_acoustic_oracle.py:131-157`, `tests/test_m6x_vertical_acoustic_oracle.py:181-195`). If ADR-023 v0 is explicitly scoped as a linearized conservative split-explicit column operator, a single tridiagonal pass is defensible as the first implementation.

The limit is that ADR-023 cites SCREAM/HOMME as its strongest production GPU evidence while omitting the thing SCREAM actually does for nonlinear pressure/exner coupling. In `DirkFunctorImpl.hpp`, SCREAM runs a Newton iteration (`E3SM/components/homme/src/theta-l_kokkos/cxx/DirkFunctorImpl.hpp:344-356`), warns on max-iteration failure (`DirkFunctorImpl.hpp:384-388`), forms an analytical tridiagonal Jacobian (`DirkFunctorImpl.hpp:707-767`), and dispatches cyclic reduction on GPU or Thomas on CPU (`DirkFunctorImpl.hpp:769-778`). That means the linear single-pass is sufficient only for the MPAS/WRF-style linearized v0. It is not justified if ADR-023 keeps advertising a SCREAM-grade nonlinear HEVI solve.

The best manager interpretation is therefore not "ADR-023 already proves the new vertical method." It is "ADR-023 selects the right experiment family." The scout's comparative table is careful about this: it found external support for clean column-implicit solves, but it also said those projects generally do not report WRF-style Tier-4 RMSE and that source-level RMSE evidence was absent for the inspected projects (`.agent/sprints/2026-05-23-m6x-dycore-alt-methods-scout/worker-report.md:7-16`). The scout recommendation also set a minimum proof of a 1-D analytic oracle plus one WRF or MPAS-derived column slice (`.agent/sprints/2026-05-23-m6x-dycore-alt-methods-scout/worker-report.md:56-63`). ADR-023 is strongest when read as ratifying that experiment, not as ratifying the final numerical method ahead of evidence.

ADR-023 is also a better project fit than ADR-022 because it admits the vertical solve is a first-class physical operator. The current `acoustic_substep_carry` only re-diagnoses pressure, applies horizontal pressure-gradient tendencies, updates u/v, optionally updates mu, then re-diagnoses pressure again (`src/gpuwrf/dynamics/acoustic_wrf.py:447-487`). There is no place where w/phi acoustic coupling is advanced, which is exactly why the oracle sees unchanged modal amplitude. ADR-022's simplified phi update would add motion, but the previous review already showed that a simplified damped-buoyancy wave equation can be finite and still physically wrong (`.agent/sprints/2026-05-22-c2-A2-A2x-bundle-review/reviewer-report.md:16-24`, `.agent/sprints/2026-05-22-c2-A2-A2x-bundle-review/reviewer-report.md:132-137`). ADR-023 is the first draft that both preserves the clean JAX contract and acknowledges the column solve has to carry the conservative couplings.

## §2 Steelman of ADR-021 vs ADR-023

The strongest case for ADR-021 is anti-tautology. A WRF-shaped port of `calc_coef_w`, `advance_mu_t`, and `advance_w` can be compared directly against WRF savepoints, and WRF is the local numerical oracle for c2 formulas under ADR-020 (`.agent/decisions/ADR-020-c2-dycore-architecture.md:20-22`). The WRF source is concrete: `advance_mu_t` builds `dvdxi`, updates `MU/MUAVE/MUTS`, computes `ww`, and then transports theta using `ww`, `fnm/fnp`, `rdnw`, and horizontal scalar terms (`/mnt/data/canairy_meteo/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1094-1173`). `advance_w` builds `t_2ave`, phi RHS, terrain bottom conditions, tridiagonal forward/back substitution, damping, and final `ph` update (`.../module_small_step_em.F:1341-1585`). If the manager's only objective were to minimize numerical-method dispute against WRF, ADR-021 would be the conservative route.

I still do not think ADR-021 is the best next decision. Its public carry expansion is large: the draft adds `t_2ave`, `ww`, `muave`, `muts`, `ph_tend`, `_1` fields, and `_save` fields to the scan carry (`.agent/decisions/ADR-021-wrf-smallstep-vertical-port-DRAFT.md:25-39`). The prior bundle review correctly identified that kind of expansion as the pivot trigger rather than a patch (`.agent/sprints/2026-05-22-c2-A2-A2x-bundle-review/reviewer-report.md:100-113`, `.agent/sprints/2026-05-22-c2-A2-A2x-bundle-review/reviewer-report.md:174-190`). ADR-021 also requires a Fortran harness before implementation (`.agent/decisions/ADR-021-wrf-smallstep-vertical-port-DRAFT.md:64-72`), which is the right proof model but not obviously the fastest path to a usable Canary operator.

WRF-baseline lateral forcing is compatible with internal MPAS-style numerics if the boundary contract is treated as a forcing/validation problem, not as an inherited timestep-algorithm requirement. The Gemini claim embedded in ADR-021 says any deviation from WRF grid nesting, hybrid eta, or split-explicit acoustic formulation will cause immediate gravity-wave blowup at the boundaries (`.agent/decisions/ADR-021-wrf-smallstep-vertical-port-DRAFT.md:17-21`). That is a useful warning, but as written it is speculative. ADR-023 is not proposing to abandon WRF-compatible metrics or boundary metadata; ADR-020 already keeps hybrid coefficients, map factors, perturbation/base-state decomposition, and WRF-shaped horizontal PGF as architecture requirements (`.agent/decisions/ADR-020-c2-dycore-architecture.md:8-10`, `.agent/decisions/ADR-020-c2-dycore-architecture.md:34-46`). The project constitution also says WRF compatibility is validation behavior and useful interfaces, not inherited architecture (`PROJECT_CONSTITUTION.md:7-10`).

The real boundary risk is narrower: if ADR-023's vertical coordinate, column mass, pressure/geopotential update, and lateral boundary application are mutually inconsistent, boundary forcing will inject gravity/acoustic noise. That risk should be tested with a WRF/MPAS-derived column slice and then a d02 boundary replay smoke. It does not force ADR-021 before ADR-023 has a chance to pass the analytic oracle plus one non-tautological savepoint/slice.

My steelman of the ADR-021 dissent is that it gives the reviewer a clearer failure surface. If a WRF savepoint exposes `t_2ave`, `ww`, `muave`, `muts`, `ph_tend`, and the `_1`/`_save` family, then a JAX port can be wrong in ways that are immediately visible at operator level. That was load-bearing in the M5 physics cycle, where real callable WRF harnesses separated real scheme work from plausible-looking substitutes. ADR-023 has to recreate that anti-tautology property with a different proof object. The analytic oracle alone is not enough: it is a flat, linear, constant-coefficient mode (`src/gpuwrf/validation/analytic_oracles/vertical_linear_acoustic.py:5-25`), while the target problem is a hybrid-eta, map-factor, base-state/perturbation column coupled into lateral boundaries and surface/physics tendencies. That is why I treat a WRF/MPAS-derived slice as mandatory before the manager uses Tier-4 language.

My steelman of ADR-023 against ADR-021 is that a line-shaped WRF port is not automatically more faithful once it is embedded in a different architecture. ADR-021's draft asks whether `_1`/`_save` families implicitly replace the current RK3 cadence with WRF's split-explicit cadence (`.agent/decisions/ADR-021-wrf-smallstep-vertical-port-DRAFT.md:91-95`). That is not a small question; it changes timestep composition and possibly state lifecycle. A partial WRF small-step port inside a non-identical outer loop could preserve more names while still failing the coupled forecast. ADR-023 avoids that by making the column solver a coherent operator under the existing JAX scan architecture, then forcing evidence to decide whether the numerical deviation is acceptable.

## §3 Stress-test ADR-023

Top three structural concerns:

1. ADR-023 currently overclaims MPAS equivalence. It should say "same conservative off-centered tridiagonal family" until a discrete derivation and savepoint/slice comparison exists.
2. ADR-023 borrows SCREAM as evidence but drops the Newton outer. That is acceptable only if v0 is explicitly a linearized MPAS/WRF-style solve, not a nonlinear SCREAM-style HEVI solve.
3. ADR-023 under-specifies the path from R7 analytic oracle to Tier-4 Gen2 RMSE and underprices the integration work needed to get there.

Findings, severity-ranked:

1. Major - The MPAS equivalence claim is not proven. ADR-023 says MPAS Klemp 2007 is mathematically equivalent to the proposed Crank-Nicolson `epssm=0.1` column solve and expects about 5 percent column-level match (`.agent/decisions/ADR-023-conservative-column-solver-DRAFT.md:76-80`). MPAS has off-centered tridiagonal structure (`.../mpas_atm_time_integration.F:1589-1591`, `.../mpas_atm_time_integration.F:2038-2041`, `.../mpas_atm_time_integration.F:2172-2208`), but its prognostic perturbation variables and coefficients differ from ADR-023's WRF-hybrid prose system. Required fix: demote this to a hypothesis and require a derivation artifact plus one MPAS or WRF column-slice comparison before any "equivalent" or fixed-percent claim is used.

2. Major - The no-Newton choice is under-justified relative to the evidence ADR-023 cites. ADR-023 says no Newton outer is needed in v0 because the system is "linear-enough" (`.agent/decisions/ADR-023-conservative-column-solver-DRAFT.md:51-56`), but SCREAM's cited implementation does the opposite: Newton iteration plus analytical tridiagonal Jacobian and failure reporting (`DirkFunctorImpl.hpp:344-356`, `DirkFunctorImpl.hpp:384-388`, `DirkFunctorImpl.hpp:707-778`). Required fix: either specify the v0 residual as a linearized MPAS/WRF acoustic-gravity solve whose unknown is only the vertically coupled linear perturbation vector, or add a Newton outer with a convergence/failure proof gate.

3. Major - The current implementation has no vertical acoustic operator for ADR-023 to replace yet, and the R7 oracle is red. The scan only applies horizontal PGF, Rayleigh damping, and optional mu continuity before returning (`src/gpuwrf/dynamics/acoustic_wrf.py:447-487`); there is no `vertical_acoustic_update` symbol, and the oracle test intentionally asserts that absence (`tests/test_m6x_vertical_acoustic_oracle.py:159-178`). The focused test run I executed produced 3 failures: quarter-period amplitude stayed 1.0 instead of approximately 0, half-period amplitude stayed 1.0 instead of -1.0, and the operator hook was absent. Required fix: R7 must go green without weakening tolerances before any coupled forecast claim.

4. Major - The solver implementation path cites a module that does not exist and blurs three solver choices. ADR-023 references `src/gpuwrf/numerics/tridiagonal_solver.py` (`.agent/decisions/ADR-023-conservative-column-solver-DRAFT.md:51-56`), but the repository has `src/gpuwrf/physics/tridiagonal_solver.py`, whose production path is `jax.lax.linalg.tridiagonal_solve` and whose Thomas recurrence is explicitly a reference/test path (`src/gpuwrf/physics/tridiagonal_solver.py:13-28`, `src/gpuwrf/physics/tridiagonal_solver.py:31-72`). The draft also says forward-sweep `lax.scan`, then says CR primitive. Required fix: pick one production API in the ADR, cite the actual module path, and profile the selected path on the target vertical/domain shape.

5. Medium - `epssm=0.1` is plausible for MPAS but not yet justified for the WRF/Gen2 target. MPAS reads `config_epssm` and uses it consistently (`.../mpas_atm_time_integration.F:1490-1496`, `.../mpas_atm_time_integration.F:1940-1942`), while WRF's formulas carry `epssm` through coefficient and update terms (`/mnt/data/canairy_meteo/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:624-646`, `.../module_small_step_em.F:1341-1345`, `.../module_small_step_em.F:1581-1585`). ADR-023 defaults to 0.1 (`.agent/decisions/ADR-023-conservative-column-solver-DRAFT.md:70-72`) but also names the parameter as an open question (`.agent/decisions/ADR-023-conservative-column-solver-DRAFT.md:108-113`). Required fix: make `epssm` a tested sweep parameter in the R7 and warm-bubble gates, with the selected default recorded from evidence.

6. Medium - The Tier-4 path is asserted rather than staged. ADR-023 says Tier-4 RMSE on `U10/V10/T2` at 24h/72h vs Gen2 is binding (`.agent/decisions/ADR-023-conservative-column-solver-DRAFT.md:79-81`), and ADR-007 says operational RMSE is the production gate (`.agent/decisions/ADR-007-precision-policy.md:30-37`). But ROADMAP M6 separately requires Tier-3 convergence, a Tier-4 small-ensemble prototype, a transfer audit, and verification-tooling preparation before M7 (`.agent/milestones/ROADMAP.md:87-98`). Required fix: add a staged acceptance ladder: R7 analytic, hydrostatic rest, WRF/MPAS column slice, warm-bubble, 1h d02 boundary replay, then 24h/72h Gen2 RMSE.

7. Medium - The `AcousticScanCarry` "no expansion" promise may still hide necessary transient state. Keeping the public carry small is good, but ADR-023 still needs local or scanned intermediates for `mu'`, theta face interpolation, column RHS, and potentially previous-stage phi/exner. The current carry has only `state`, `previous_pressure`, `al`, `alt`, `cqu`, and `cqv` (`src/gpuwrf/dynamics/acoustic_wrf.py:43-76`), while WRF's canonical path depends on `ww`, `muave`, `muts`, `ph_tend`, and time-averaged theta (`/mnt/data/canairy_meteo/artifacts/wrf_gpu_src/WRF/dyn_em/module_small_step_em.F:1102-1114`, `.../module_small_step_em.F:1341-1345`). Required fix: explicitly distinguish public scan carry, per-substep locals, and solver scratch so "no expansion" does not become "missing physics".

8. Medium - The manager's 3-5 worker-day estimate is optimistic. ADR-023 estimates 3-5 days to warm-bubble pass (`.agent/decisions/ADR-023-conservative-column-solver-DRAFT.md:85-92`), but the parallel prototype report is still an empty stub (`.agent/sprints/2026-05-23-m6x-adr023-conservative-column-prototype/worker-report.md:1-12`), and the current oracle is fully red (`.agent/sprints/2026-05-23-m6x-vertical-acoustic-analytic-oracle/worker-report.md:20-37`). Required fix: replan around proof gates, not calendar optimism.

9. Medium - The draft does not yet say how the conservative solve preserves or updates pressure consistently with `diagnose_pressure_al_alt`. The existing code diagnoses pressure from theta/alt and geopotential/mass when a base state is present (`src/gpuwrf/dynamics/acoustic_wrf.py:188-219`). ADR-023 says it solves over `(w, mu, theta, phi/exner)` (`.agent/decisions/ADR-023-conservative-column-solver-DRAFT.md:34-45`), but it does not specify which fields are prognostic outputs, which are diagnostics, and whether pressure is updated by EOS, exner, or the existing diagnostic pathway. Required fix: define post-solve replacement order for `w`, `theta`, `ph_perturbation`, `mu_perturbation`, `p_perturbation`, `al`, and `alt`.

10. Medium - Performance and residency claims are not yet proof-backed. The constitution and roadmap require no timestep-loop host/device transfers and profiler artifacts for performance claims (`PROJECT_CONSTITUTION.md:6`, `.agent/milestones/ROADMAP.md:124-128`). ADR-023 correctly repeats the no-transfer constraint (`.agent/decisions/ADR-023-conservative-column-solver-DRAFT.md:74-81`), but a new tridiagonal solve can introduce shape polymorphism, host fallback, or extra launches if not staged carefully. Required fix: add a transfer audit and HLO/profiler proof to the implementation sprint, even before full 24h RMSE.

## §4 Cost re-estimation

I would re-estimate ADR-023 in two tiers.

Tier A, "operator proof": 5-8 focused worker-days plus reviewer/tester cycles. That includes one day to freeze the discrete equations and variable mapping; one to two days to build the coefficient/RHS builder with WRF hybrid denominators and `epssm`; one to two days to integrate the actual JAX tridiagonal solve and expose `vertical_acoustic_update`; one day to make the three R7 tests pass; and one to two days for transfer/static checks, warm-bubble replay, and review fixes. This assumes no Newton outer and no MPAS harness. If the derivation shows nonlinear exner/EOS coupling is necessary, add 2-4 days for Newton residual/Jacobian work and failure handling.

Tier B, "forecast relevance": another 3-6 worker-days after Tier A. A 24h/72h Tier-4 RMSE run cannot honestly start from the current state because the operator hook is absent and the R7 gate is red. Before Gen2 RMSE, the project needs a non-tautological column slice, warm-bubble recovery, transfer audit, and at least a 1h d02 boundary replay. The earliest honest Tier-4 RMSE-vs-Gen2 run is after one successful prototype sprint and one integration/forecast sprint. If the prototype produces only analytic-oracle success, the RMSE run should be labeled exploratory, not a closeout gate.

The performance cost is also not priced yet. ADR-023's `lax.scan` Thomas path is simple but serial along vertical levels; `jax.lax.linalg.tridiagonal_solve` may lower to a better GPU primitive, but the repository's reusable solver is in `src/gpuwrf/physics/tridiagonal_solver.py`, not the path cited by ADR-023 (`src/gpuwrf/physics/tridiagonal_solver.py:13-28`). SCREAM uses CR on GPU and Thomas on CPU (`DirkFunctorImpl.hpp:769-778`), which is exactly the performance distinction ADR-023 must measure rather than assume. A credible prototype needs HLO or profiler evidence for launch count, transfer count, and solver wall time on the target d02 vertical shape.

A practical sprint sequence would be:

1. ADR patch and interface freeze. Edit only the ADR/spec, not model code. Outputs: equation block, variable ownership, solver API, acceptance ladder, fallback trigger.
2. Operator prototype. Implement a pure function that takes one batched column state plus metrics and returns updated `w`, `theta`, `ph_perturbation`, `mu_perturbation`, and pressure diagnostics. Outputs: R7 green, hydrostatic rest green, no transfer primitives, no production dtype downgrade.
3. Non-tautological slice. Build either an MPAS column extraction from the local source path or a WRF small-step diagnostic slice. Outputs: comparison JSON and a manager decision on whether the mismatch is acceptable.
4. Integration smoke. Wire the operator into `acoustic_substep_carry`, run warm-bubble and a short d02 boundary replay. Outputs: finite/stability proof, transfer audit, launch/solver profile.
5. Forecast comparison. Only then run 24h/72h Gen2 RMSE. Outputs: operational comparison, not just "operator works."

This is why I do not believe the 3-5 day estimate for a meaningful Tier-4 answer. Three to five days may be enough for first R7 green if the single-pass linearized solver is right. It is not enough for a defensible Gen2 RMSE verdict unless the prototype worker returns a nearly complete implementation and the existing coupled driver accepts the new operator without new boundary, dtype, or launch-count regressions.

## §5 Verdict

RATIFY-ADR-023

Ratification should be narrow: ADR-023 is the best direction, but the manager should revise the draft before treating it as an implementation contract. The revision should remove the unproven MPAS equivalence claim, define v0 as either linearized MPAS/WRF-style or nonlinear SCREAM-style, fix the tridiagonal solver path, and add the staged proof ladder from analytic oracle to Gen2 RMSE. If the R7 oracle or the first WRF/MPAS column slice fails, fallback should be ADR-021's WRF-shaped port, not ADR-022's simplified phi update.

Dissent: I would not call this an acceptance of the current ADR-023 text. The text is still too confident in three places: the MPAS equivalence language, the no-Newton justification, and the schedule. A strict reviewer could reasonably reject the draft until those are patched. I am choosing ratification because the decision before this critic is architectural direction among three options, and ADR-023 is the only option that simultaneously respects the project's non-line-by-line architecture, the small carry boundary, and the evidence that the vertical solve must be conservative. The ratification should be recorded with required fixes, not as permission to ship the draft verbatim.

The rejected alternative I feel most strongly about is ADR-022 as written. Its omission of mass/geopotential weighting was already called out as unsupported by the scout and review evidence (`.agent/sprints/2026-05-23-m6x-dycore-alt-methods-scout/worker-report.md:38-44`, `.agent/sprints/2026-05-22-c2-A2-A2x-bundle-review/reviewer-report.md:38-44`). Keeping that route would be betting Tier-4 RMSE can hide a known operator-level physics gap. That is a poor bet in a project whose constitution says physics correctness precedes speed claims (`PROJECT_CONSTITUTION.md:5`).

## §6 Open questions

1. What exact residual is ADR-023 solving? The draft needs a discrete equation block that names the unknown vector, old/new terms, explicit terms, and the terms eliminated into the tridiagonal coefficients.

2. Is v0 intentionally linearized? If yes, stop citing SCREAM as proof for omitting Newton; cite SCREAM only as proof that nonlinear HEVI requires Newton/Jacobian machinery. If no, add Newton from day one.

3. What is the first non-tautological comparison target after the analytic oracle: a WRF small-step savepoint, an MPAS column slice, or both? The scout recommended at least one WRF or MPAS-derived column slice (`.agent/sprints/2026-05-23-m6x-dycore-alt-methods-scout/worker-report.md:56-63`).

4. Which tridiagonal implementation is production: `jax.lax.linalg.tridiagonal_solve`, Thomas via `lax.scan`, or a custom CR/Pallas/Triton kernel? The answer affects both performance and the proof object.

5. What `epssm` values will be swept before default selection? I would test 0.0, 0.1, and the WRF/Gen2 configured value, then bind the default to R7 plus warm-bubble plus one d02 smoke result.

6. What is the fallback trigger? I recommend: if ADR-023 cannot pass R7 plus one WRF/MPAS column slice within one prototype sprint plus one fix sprint, stop and dispatch the ADR-021 WRF-shape harness path.
