# M6 Solver Comparison: Thomas vs PCR vs Batched-Thomas

## Decision Context

This scout supports ADR-026 / M6-perf-design solver selection for the WRF-shaped vertical-implicit acoustic solve. The validated M6B2 implementation is the WRF serial Thomas forward/back recurrence in `src/gpuwrf/dynamics/tridiag_solve.py`, with savepoint parity passing on column, 16x16 patch, and golden d02 tiers. PROJECT_PLAN 14.5.1 keeps this distinction binding: validation mode follows WRF's serial Thomas boundary, while operational mode may choose Thomas, parallel cyclic reduction (PCR), or a batched / hybrid solver only after profiler evidence and Tier-4 RMSE evidence.

Canary d02 is `west_east=159`, `south_north=66`, `bottom_top=44`, so it has 10,494 independent vertical columns and 461,736 unstaggered cells. The sprint shorthand "10,500 columns" is acceptable for occupancy estimates. The 1 km target shorthand "480x200x60 = 5.76M columns" should be read as 5.76M cells, not columns; it is 96,000 columns with 60 vertical unknowns per column.

Public sources used here are deliberately conservative. Zhang, Cohen, and Owens' NVIDIA/UC Davis paper gives the most relevant GPU tridiagonal algorithm cost model: Thomas is simple but serial, CR/PCR/RD trade more arithmetic for fewer dependency steps, and CR+PCR hybrids can outperform standalone methods on GPUs because performance depends on memory access, synchronization/control, bank conflicts, and arithmetic together, not on FLOPs alone (https://research.nvidia.com/sites/default/files/pubs/2010-01_Fast-Tridiagonal-Solvers/Zhang_Fast_2009.pdf). NVIDIA cuSPARSE documents production `gtsv2` / `gtsv2_nopivot` tridiagonal routines, including pivoting tradeoffs and a CR+PCR no-pivot implementation (https://docs.nvidia.com/cuda/cusparse/index.html). NVIDIA cuSolverDx explicitly states that its device `gtsv_no_pivot` uses a PCR + Thomas hybrid because Thomas is efficient but sequential and PCR is more expensive but parallelizable (https://docs.nvidia.com/cuda/cusolverdx/0.4.0/get_started/functions/gtsv.html).

RTX 5090 context matters. NVIDIA's Blackwell architecture brief lists GeForce RTX 5090 as GB202 with 170 SMs, 21,760 CUDA cores, 32 GB GDDR7, and 1,792 GB/s memory bandwidth (https://images.nvidia.com/aem-dam/Solutions/geforce/blackwell/nvidia-rtx-blackwell-gpu-architecture.pdf). The memory bandwidth is large enough that this solve will probably be control/launch/temporary limited before it is pure DRAM-bandwidth limited at d02, but that is an estimate, not measured.

## Suitability for WRF Vertical-Implicit Acoustic

WRF's source describes `advance_w` as advancing the implicit w and geopotential equations (https://www2.mmm.ucar.edu/people/wrfhelp/wrfvar/code/trunk/html/module_small_step_em.html). MPAS-A's technical note gives a closely related atmospheric precedent: eliminating thermodynamic and density perturbations in the vertically implicit acoustic step yields a tridiagonal system for vertical velocity, with upper/lower boundary conditions and back-substitution for other acoustic variables (https://www2.mmm.ucar.edu/projects/mpas/mpas_website_linked_files/MPAS-A_tech_note.pdf). The solver property required by the physics is therefore: solve the same vertically implicit tridiagonal system accurately enough that the acoustic step, damping, and back-substitution stay inside the forecast envelope. It does not require the Thomas recurrence as a mathematical property, but WRF parity currently does require it in validation mode.

The main solver-specific correctness difference is floating-point order and pivoting. WRF's Thomas path is the local reference. PCR and hybrids change operation order and usually run without pivoting for performance. Zhang et al. state that no-pivot GPU solvers may fail on general tridiagonal matrices, while CR/PCR are stable without pivoting for diagonally dominant or symmetric positive definite matrices; their experiments show good residuals for diagonally dominant fluid matrices but worse behavior for non-dominant cases (same NVIDIA paper URL above). The WRF acoustic coefficients must therefore be checked empirically for diagonal dominance / pivots on representative Canary columns, then validated by Tier-4 RMSE. A solver switch is operational-mode-only unless WRF itself uses it, which it does not in this validated path.

ICON and GT4Py prior art supports the broader GPU rewrite strategy but does not prove PCR is the right local solver. MeteoSwiss' operational ICON GPU paper reports 5.5x socket-to-socket speedup after whole-model GPU porting, mixed precision, and performance tuning, with meteorological quality constraints met (https://gmd.copernicus.org/articles/19/755/2026/index.html). ICON-EXCLAIM's Python/GT4Py dycore paper reports the GT4Py dycore about 10% faster than Fortran+ near the "just-fits" GPU configuration and explicitly warns that strong scaling hits an occupancy asymptote as per-GPU work decreases (https://gmd.copernicus.org/articles/19/713/2026/index.html). This is directly relevant to d01: fewer columns can make vertical parallelism more attractive, but for large d02/1km domains column batching is already substantial.

GT4Py's own quickstart uses a Thomas tridiagonal solver example expressed as forward and backward vertical computations, showing that weather DSLs commonly represent this as ordered vertical loops rather than PCR by default (https://gridtools.github.io/gt4py/latest/quickstart.html). Pace is an FV3GFS/SHiELD Python implementation using GT4Py that can run on heterogeneous nodes and CUDA backends, which is useful precedent for DSL-based atmospheric kernels, but I did not find public Pace evidence that its vertical solve mandates PCR for this WRF acoustic case (https://github.com/ai2cm/pace).

JAX compatibility is reasonable for all three in principle. `jax.lax.scan` is a JAX primitive lowered to a single WhileOp, useful for reducing compilation size compared with Python-loop unrolling, and it requires fixed carry shape/dtype (https://docs.jax.dev/en/latest/_autosummary/jax.lax.scan.html). JAX's async-dispatch docs warn that host inspection or conversion blocks; real timings need `block_until_ready()` and operational code must avoid host value reads (https://docs.jax.dev/en/latest/async_dispatch.html). Transfer guard docs identify explicit `device_get` / `device_put` and implicit transfers as guardable events (https://kolonist26-jax-kr.readthedocs.io/en/latest/transfer_guard.html). Current JAX install docs say CUDA 12 wheels support NVIDIA GPUs with SM >= 5.2 and CUDA 13 wheels support SM >= 7.5, with CUDA 13 migration recommended; that is compatible with Blackwell-class deployment, subject to the repo's pinned wheel and driver audit (https://docs.jax.dev/en/latest/installation.html).

## Expected Speed At d02, d01, and 1 km

All speedups here are estimates, not measured. They should not be promoted to ADR facts without Nsight Systems / Nsight Compute artifacts.

For d02, current batched Thomas is the default favorite. There are ~10,500 columns, so the solver has enough horizontal batch to expose work even though each column has a serial vertical dependency. PCR reduces recurrence depth from ~44 to 6 rounds but increases arithmetic by roughly 9x under Zhang's scalar op model (12nlog2n vs 8n) and creates more temporary coefficient traffic in JAX. Expected winner: **Thomas or batched-Thomas**, unless profiling shows the `lax.scan` WhileOp body/control overhead dominates. Plausible speedup from PCR/hybrid is 0.8x to 1.8x versus current Thomas, not a justified switch. Bottleneck estimate: scan loop/control overhead and temporary memory traffic, not raw FP32 throughput.

For d01, 50x50x44 gives 2,500 columns and 110,000 cells. If one thread owns one column, that is only ~78 warps of column work, below the RTX 5090's 170 SM count. Vertical parallelism from PCR could matter more here, but kernel launch/control overhead and extra arithmetic become proportionally more visible. Expected winner: **uncertain; hybrid/PCR deserves a bakeoff**. Bottleneck estimate: occupancy and launch overhead first, memory traffic second.

For the 1 km target, 480x200x60 is 96,000 columns and 5.76M cells. Column parallelism is abundant. PCR's extra arithmetic is hard to justify unless Nsight shows the serial vertical recurrence is the dominant whole-timestep hotspot after fusion. Expected winner: **batched Thomas / current Thomas shape**, possibly with operational fusion and mixed precision, not full PCR. Bottleneck estimate: memory bandwidth, XLA temporary allocation, and whole-timestep fusion; not insufficient vertical parallelism.

## Implementation Cost

| Algorithm | Estimated LOC in JAX | Testing surface | Risk |
|---|---:|---|---|
| Thomas via `lax.scan` | 0 for validation helper; perhaps 20-60 LOC to expose an operational fused entry if ADR-026 keeps it. | Existing M6B2 parity plus operational Tier-4, HLO/Nsight no-transfer proof. | Low. Reference path, but speed ceiling may disappoint. |
| PCR via `lax.scan` | 120-180 LOC for padded non-power-of-two coefficient updates, masks, boundary rows, and solution extraction. | n=44 non-power-of-two; n=45 staggered; top/bottom boundaries; near-zero pivots; diagonal-dominance diagnostics; Thomas equivalence; Tier-4 RMSE. | Medium/high. More temporary traffic and different rounding. |
| Batched-Thomas via `vmap` + scan | 0-40 LOC because current helper already batches over trailing dimensions; 80-150 LOC if reshaping to explicit column-major operational layout. | Same as Thomas plus layout and fusion checks. | Low/medium. Best near-term operational baseline. |
| PCR+Thomas hybrid | 160-260 LOC in JAX for one or two fixed split sizes; 200-500 LOC if benchmarking cuSolverDx/cuSPARSE through FFI/custom call. | All PCR tests plus split-size selection, padding, and profiler artifacts. | Medium. Best candidate for bakeoff, not for blind switch. |

## Recommendation

**BAKEOFF.** M6-perf-design should not commit to PCR from literature alone. The next small implementation sprint should benchmark three operational-only candidates on the same d02/d01/1km-shape synthetic coefficient tensors and at least one real Canary coefficient savepoint: current batched Thomas, pure PCR, and one fixed PCR+Thomas hybrid. The bakeoff must collect HLO text/cost analysis, Nsight Systems no-H2D/D2H proof, kernel/loop counts, wall time with `block_until_ready()`, and Thomas-vs-candidate residuals before any Tier-4 forecast run. cuSPARSE/cuSolverDx should be benchmark references only unless ADR-026 explicitly accepts JAX FFI/custom calls.

The practical default for ADR-026 before bakeoff should be KEEP-THOMAS for d02/1km operational design, with BATCHED-THOMAS treated as the same implementation family, because d02 already has enough independent columns and M6B2 has correctness proof. PCR is more attractive for d01 or for a profiler-demonstrated vertical recurrence hotspot, not as an assumption.

Dissent: PCR may still win. The RTX 5090 has enough SMs that a short serial recurrence over only 10,500 columns may leave issue slots underused, especially if XLA emits a separate kernel body per scan step or fails to fuse adjacent acoustic work. NVIDIA's own library direction uses PCR+Thomas hybrids for device tridiagonal solves, which is strong evidence that a hybrid can beat plain Thomas for some problem sizes. If the M6-perf-design hotspot is the scan loop rather than memory traffic, skipping PCR would leave a real speedup on the table.
