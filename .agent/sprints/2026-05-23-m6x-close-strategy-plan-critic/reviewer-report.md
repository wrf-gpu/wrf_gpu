# Reviewer Report - M6 Close Strategy Plan Critic

## §1 Critique of the Manager's Draft Plan

The manager's draft has the right governing instinct: stop treating the unsourced warm-bubble amplitude as the architecture gate and measure the documented M6 gates first. That is supported by the current project state: the morning report says the old `[5, 10] m/s` warm-bubble target is being replaced because both apparent passes used unphysical stabilizers and M6's actual close gate is Tier-3 plus initial Tier-4 consistency (`MORNING-REPORT.md:22-29`, `MORNING-REPORT.md:76-89`). It is also aligned with the validation strategy, where Tier 3 is short-run convergence and Tier 4 is ensemble/statistical consistency, not one synthetic amplitude (`VALIDATION_STRATEGY.md:13-19`). The problem is execution order and measurable-goal quality. The plan says "measure first, fix after" (`manager-plan-m6-close-strategy.md:15`), but S2 makes a source-derived operator change in parallel with S1 before the d02 measurements exist (`manager-plan-m6-close-strategy.md:34-43`). That is a real contradiction.

**S1 - d02 1h boundary replay + sidecars.** The structure is directionally right because the halted d02 replay has real scaffolding and was explicitly preserved for redispatch (`.agent/SPRINT-TRACKER.md:21-22`). The current helper already loads Gen2 d02 hourly side-history boundaries (`src/gpuwrf/integration/d02_replay.py:142-206`), builds a Gen2 d02 initial state preserving P/PB, PH/PHB, and MU/MUB splits (`src/gpuwrf/integration/d02_replay.py:209-291`), runs RK3 stages plus acoustic scans (`src/gpuwrf/integration/d02_replay.py:353-385`), applies physics and lateral boundaries (`src/gpuwrf/integration/d02_replay.py:388-422`), and compares surface/column fields to Gen2 (`src/gpuwrf/integration/d02_replay.py:515-546`). That makes S1 the right first proof rung.

What is missing is a hard status definition beyond "completes without nonfinite." The replay payload currently marks PASS if post-sanitize state is finite, static callbacks are absent, and trace transfers are zero (`src/gpuwrf/integration/d02_replay.py:763-768`), while RMSE is explicitly informational (`src/gpuwrf/integration/d02_replay.py:788-792`). Because the replay loop sanitizes candidates before emitting diagnostics (`src/gpuwrf/integration/d02_replay.py:450-451`), S1 must fail on pre-sanitize candidate nonfinite or clip counts, not only final finite state. The manager's RMSE table at 15, 30, and 60 minutes is also not yet measurable unless Gen2 output exists at those lead times; the loader's reference lookup is hourly-indexed (`src/gpuwrf/integration/d02_replay.py:495-512`). If only hourly wrfout is present, 15 and 30 minute RMSE require either interpolated truth or explicit "diagnostic only, no Gen2 truth" status.

**S2 - MPAS damping derivation.** This is the weakest sprint as written. Removing `_mu_continuity_increment` is necessary before ADR-023 can be accepted because the code itself says it is a temporary validation stabilizer, not an accepted 24h mechanism (`src/gpuwrf/dynamics/acoustic_wrf.py:456-475`), and the unified worker says the same risk remains (`.agent/sprints/2026-05-23-m6x-adr023-public-scan-path-unification/worker-report.md:50-54`). But the proposed replacement is not actually a same-equation replacement. MPAS's cited `dss` block is implicit Rayleigh damping on `rw_p` after the tridiagonal solve (`mpas_atm_time_integration.F:2184-2192`); `_mu_continuity_increment` limits dry-column mass update after `mu_continuity_tendency` (`src/gpuwrf/dynamics/acoustic_wrf.py:872-876`). Those are different variables and different failure modes.

The measurable goals also overclaim. "Replace with MPAS-line-cited dss-style damping" (`manager-plan-m6-close-strategy.md:38-40`) is not measurable unless the sprint defines the equation, target variable, coefficient source, and validation artifact. The same applies to removing the `0.38` and `1.35` constants. The current code exposes both constants (`src/gpuwrf/dynamics/acoustic_wrf.py:31-33`), and Opus correctly found that `1.35` is not the MPAS geometric conversion for a real stratified column (`.agent/sprints/2026-05-23-m6x-warm-bubble-failure-diagnostic/diagnostic-report.md:124-125`). MPAS diagnoses `w` from `rw` by dividing by a layer-dependent `zz` interpolation (`mpas_atm_time_integration.F:2491-2495`), so replacing `1.35` with a per-level metric is plausible. But the sprint should be a source-audit plus A/B experiment after S1 baseline, not parallel production surgery.

**S3 - 24h Tier-4 RMSE.** Extending from 1h to 24h is the right direction because ADR-023 says Tier-4 RMSE on U10/V10/T2 at 24h/72h vs Gen2 is the binding gate (`.agent/decisions/ADR-023-conservative-column-solver.md:100-107`), and ADR-007 makes operational RMSE the production gate for precision decisions (`.agent/decisions/ADR-007-precision-policy.md:30-37`). The proposed fields T2/U10/V10 are correct. The proposed acceptance threshold is not yet defensible. "RMSE < 2x Gen2 day-over-day spread" (`manager-plan-m6-close-strategy.md:49-53`) is not in ADR-007, which instead binds U10/V10/T2 to CPU-vs-observation noise floor over a representative Canary sample (`.agent/decisions/ADR-007-precision-policy.md:36`). The prior M6 critique also rejected loosened operational gates that can hide GPU-vs-CPU error behind broad tolerance (`.agent/sprints/2026-05-21-m6-milestone-plan-scout/critical-review-codex.md:155-175`). If observations are not available, 24h can produce a provisional Gen2-consistency diagnosis, but it should not close M6 as operationally green.

**S4 - Tier-3 short-run convergence.** Tier-3 should come before 24h Tier-4, not parallel with it. The validation strategy defines Tier-3 as short forecast windows and timestep convergence checks with drift inside a documented envelope (`VALIDATION_STRATEGY.md:13-15`). The earlier M6 plan review was explicit that a controlled dt-refinement case is required and that using different Gen2 domains or backfill campaigns as a "CPU envelope" risks false confidence (`.agent/sprints/2026-05-21-m6-milestone-plan-scout/critical-review-codex.md:54-59`, `.agent/sprints/2026-05-21-m6-milestone-plan-scout/critical-review-codex.md:151-153`). The manager's "sub-quadratic dt-doubling growth" (`manager-plan-m6-close-strategy.md:57-60`) is too vague: it does not name norms, variables, accepted lead windows, base/refined dt pairs, boundary mode, or whether the comparison is GPU self-convergence, WRF envelope, or both.

**S5 - closeout.** The decision structure is clear but too optimistic. Promoting ADR-024 is plausible if the manager accepts the gate change; ADR-024 already has proof objects and explicitly says warm-bubble is an operator-sanity gate (`.agent/decisions/ADR-024-warm-bubble-gate-policy.md:15-24`, `.agent/decisions/ADR-024-warm-bubble-gate-policy.md:47-52`). Promoting ADR-023 to ACCEPTED on S3/S4 pass alone is unsafe if `_mu_continuity_increment` remains, because ADR-024 does not ratify any operator architecture or remove that limiter (`.agent/decisions/ADR-024-warm-bubble-gate-policy.md:41-45`). ADR-023 itself still says reviewer concurrence is required and that the public warm-bubble path exposed unresolved mu-stabilization risk (`.agent/decisions/ADR-023-conservative-column-solver.md:1-3`, `.agent/decisions/ADR-023-conservative-column-solver.md:163-175`).

**Cross-cutting inspiration mining and risk register.** The source list is good and should stay. WRF has the specific time-averaged `MUAVE`, `ww`, and `t_2ave` machinery the current path lacks (`module_small_step_em.F:1102-1119`, `module_small_step_em.F:1341-1345`, `module_small_step_em.F:1451-1489`). MPAS has the off-centered recurrence, tridiagonal solve, Rayleigh damping, `wwAvg` accumulation, and density/theta reconstruction in one canonical region (`mpas_atm_time_integration.F:2141-2208`). Pace and ICON4Py show that production dycores keep vertical solvers, damping, and hyperdiffusion as named scheme components, not as hidden clamp constants (`Pace@6a46e69:fv3core/pace/fv3core/stencils/dyn_core.py:472-574`, `ICON4Py@3934f68:model/atmosphere/dycore/src/icon4py/model/atmosphere/dycore/solve_nonhydro.py:139-194`). The manager's risk register misses the most important structural risk: neither ADR-023 nor clamp-stripped ADR-021 may pass Tier-3/Tier-4 within budget, in which case the project needs an explicit exit rule, not another architecture debate.

## §2 My Version Of The Plan

I recommend a six-sprint hybrid. It keeps the manager's measure-first premise, but serializes baseline measurement before operator changes and makes Tier-3 a gate before 24h Tier-4. It also turns source mining into a bounded proof object, not background advice.

**S1 - Diagnostic sidecar foundation and source-mining lock.** Goal: build the diagnostic outputs and source-mining table before changing the operator. Acceptance: `diagnostic_field_rmse_timeline.py`, `diagnostic_spatial_divergence_map.py`, `diagnostic_conservation_tracker.py`, `diagnostic_bound_violation_tracer.py`, `diagnostic_sanitizer_audit.py`, and `diagnostic_limiter_activation.py` exist; each writes JSON with field names, units, shape, lead time, and status; `source_mining_operator_table.md` cites WRF/MPAS/Pace/ICON4Py/Dinosaur lines for every operator term changed later; no production `src/gpuwrf` edit. Time budget: 6-10 hours. Dependencies: none.

Rationale: This sprint is cheap insurance. The current d02 replay proof already says RMSE is informational and sanitization is separate from finiteness (`src/gpuwrf/integration/d02_replay.py:788-792`), so the project needs sidecars before a long run becomes decision evidence. The warm-bubble redesign shows that improved reporting changed the verdict from a hidden amplitude failure to `FAIL_PHYSICAL_BOUNDS` (`.agent/sprints/2026-05-23-m6x-warm-bubble-gate-redesign/worker-report.md:21-35`, `.agent/sprints/2026-05-23-m6x-warm-bubble-gate-redesign/worker-report.md:50-52`). The source-mining lock prevents another unreviewed "0.38/1.35" path: MPAS uses `zz`-weighted conversion for `w` (`mpas_atm_time_integration.F:2491-2495`), WRF's buoyancy term uses `t_2ave` and `muave` (`module_small_step_em.F:1451-1489`), ICON4Py names Rayleigh and divergence damping configuration (`ICON4Py@3934f68:model/atmosphere/dycore/src/icon4py/model/atmosphere/dycore/solve_nonhydro.py:182-194`), and Dinosaur shows the clean IMEX separation pattern (`Dinosaur@59a0197:dinosaur/time_integration.py:74-114`).

**S2 - Current ADR-023 1h d02 replay baseline, no operator changes.** Goal: run the current unified ADR-023 path through the 1h Gen2 d02 replay and produce a diagnosis, not a fix. Acceptance: `scripts/m6_d02_boundary_replay_1h.py --duration-s 3600 --dt-s 1 --n-acoustic 4` produces a proof JSON; `first_nonfinite_step` and `first_candidate_nonfinite_step` are both null; `candidate_clip_count_total == 0` or is explicitly classified as `FAIL_SANITIZER_MASKING`; transfer trace post-init bytes are zero; RMSE and max-error are reported for T2, U10, V10, `w_k20`, and `theta_k20` at the available Gen2 lead; boundary metadata names the run directory and side-history schema. Time budget: 8-14 hours. Dependencies: S1 diagnostic schemas.

Rationale: This is the manager's S1, narrowed to a trustworthy baseline. It respects the evidence that the replay scaffold is ready (`src/gpuwrf/integration/d02_replay.py:706-794`) and that boundary forcing is built from Gen2 d02 hourly side histories (`src/gpuwrf/integration/d02_replay.py:142-206`). It also respects the current risk: the replay loop sanitizes the candidate (`src/gpuwrf/integration/d02_replay.py:450-451`), so finiteness after sanitization cannot be the only pass criterion. This sprint answers the plan's most important question first: does the current path fail immediately on a real-ish coupled d02 replay, and where? If S2 fails because sanitizer or limiter activity is masking instability, S3 becomes a targeted operator sprint. If S2 is finite with tolerable 1h errors, S3 can still remove the temporary stabilizer, but it will do so with baseline deltas rather than speculation.

**S3 - Mu/metric operator cleanup as an A/B experiment.** Goal: replace or ratify the temporary mu limiter and derive the MPAS metric constants only after S2 baseline evidence exists. Acceptance: `_mu_continuity_increment` is either removed or gated behind an explicit `NOT_ACCEPTED_FOR_M6_CLOSE` status; any replacement cites WRF/MPAS equations and is tested against warm-bubble operator sanity plus the S2 1h d02 baseline; `MPAS_OMEGA_TO_W_METRIC` is replaced by a per-level metric derived from MPAS `rw -> w` geometry or documented as slice-only; `MPAS_COLUMN_BUOYANCY_TENDENCY_SCALE` is not used as production physics unless a source or oracle pins it; all R7, hydrostatic rest, MPAS slice, path-unification, and pressure-wiring tests stay green. Time budget: 8-16 hours. Dependencies: S1 source table and S2 baseline.

Rationale: This is the manager's S2, but with the right evidence boundary. MPAS `dss` is legitimate Rayleigh damping on vertical momentum (`mpas_atm_time_integration.F:2184-2192`), and WRF has a top sponge form in `advance_w` (`module_small_step_em.F:1559-1569`), but neither is a dry-column mass-continuity limiter. Therefore the sprint must not pretend "dss-style damping" is a drop-in fix for `_mu_continuity_increment`. It should separately fix metric realism and mu stability. MPAS computes coefficients with `zz`, pressure, theta, and density factors (`mpas_atm_time_integration.F:1589-1656`) and later converts `rw` to `w` through `zz` interpolation (`mpas_atm_time_integration.F:2491-2495`). The current constant metric (`src/gpuwrf/dynamics/acoustic_wrf.py:721-758`) is therefore a test artifact until proven otherwise. Acceptance: `artifacts/m6/tier3/tsc_envelope.json` records base/refined dt pairs, case definition, boundary mode, forcing mode, variables, norms, lead times, and pass/fail; run at least `dt`, `dt/2`, and either `dt/4` or an accepted CPU/WRF envelope; require no nonfinite, no sanitizer masking, clean transfer audit, and bounded drift for U/V/W/theta/p/mu; separate "TSC" from "Gen2 backfill consistency." Time budget: 8-14 hours. Dependencies: S3 if S2 exposed limiter instability; otherwise can start after S2 baseline.

Rationale: Tier-3 is the first binding correctness gate after isolated fixtures, and the previous M6 critical review already warned that configuration-noise comparisons are not timestep convergence (`.agent/sprints/2026-05-21-m6-milestone-plan-scout/critical-review-codex.md:54-59`). The manager's S4 is right in spirit but too underspecified. A controlled convergence case should be small enough to run repeatedly, with identical IC/BC and physics scope. If the current operator cannot show convergence on this, a 24h forecast can at best be an expensive failure trace. This also forces the dycore question back into evidence: WRF's RK machinery and acoustic substep cadence are explicit (`module_em.F:670-733`), MPAS uses a forward-backward vertically implicit step (`mpas_atm_time_integration.F:2141-2208`), and Dinosaur shows a JAX-friendly explicit/implicit split with solver inverse semantics (`Dinosaur@59a0197:dinosaur/time_integration.py:74-114`, `Dinosaur@59a0197:dinosaur/time_integration.py:329-405`).

**S5 - 6h then 24h Tier-4 Gen2/observation comparator.** Goal: run operational-surface comparison only after the operator passes short-run convergence. Acceptance: produce 6h and 24h proof JSONs for T2, U10, V10, qv2/Q2 if available, `w_k20`, and theta; include land/sea/elevation stratification; compare GPU-vs-Gen2 to CPU-vs-observation noise floor where observations exist; if observations are unavailable, close as `PROVISIONAL_VALIDATION` only; record Gen2 sample/spread as secondary context, not a loosening factor; include transfer audit and peak memory. Time budget: 12-24 hours. Dependencies: S4 pass.

Rationale: This preserves the manager's S3 but fixes the threshold. ADR-007's production gate is operational RMSE against the CPU-vs-observation noise floor for U10/V10/T2 (`.agent/decisions/ADR-007-precision-policy.md:36`), while MILESTONES says M6 needs Tier-3 and initial Tier-4 consistency (`MILESTONES.md:53-60`). Gen2 day-over-day spread may be a useful diagnostic if observations are missing, but it cannot ratify operational quality alone. A 6h checkpoint should precede 24h because it localizes failures and reduces wasted runtime. Pace and ICON4Py both treat damping, hyperdiffusion, vertical solves, and physics-adjacent correction as named components rather than hidden afterthoughts (`Pace@6a46e69:fv3core/pace/fv3core/stencils/dyn_core.py:557-574`, `ICON4Py@3934f68:model/atmosphere/dycore/src/icon4py/model/atmosphere/dycore/solve_nonhydro.py:1289-1378`); the comparator should reveal which component dominates error growth.

**S6 - M6 closeout or explicit architecture exit.** Goal: make a decision, not start another open-ended dycore loop. Acceptance: write `.agent/decisions/MILESTONE-M6-CLOSEOUT.md` if S4/S5 pass and no temporary stabilizer remains in the acceptance path; promote ADR-024 to ACCEPTED if the gate policy is ratified; promote ADR-023 only if the mu limiter is removed/ratified by evidence and reviewer concurrence exists; otherwise write `M6-DYCORE-BLOCKER.md` with one of three next actions: clamp-stripped ADR-021 WRF-small-step sprint, ADR-023-plus-WRF-scratch hybrid sprint, or third-path substrate scout. Time budget: 2-4 hours. Dependencies: S4/S5 or an early structural fail.

Rationale: The current project has already run enough architecture sprints to need an exit rule. ADR-023 is still PROPOSED and explicitly requires reviewer concurrence (`.agent/decisions/ADR-023-conservative-column-solver.md:1-3`). ADR-021 has correct WRF-shape intent but its prototype pass was contaminated by clamps (`MORNING-REPORT.md:20-21`, `.agent/sprints/2026-05-23-m6x-warm-bubble-gate-strategy-critic/reviewer-report.md:21`). The closeout sprint must not declare success if it merely shifts the instability into a limiter or sanitizer. If the evidence fails, the next decision should be forced into a bounded choice. WRF small-step has real time-averaged scratch terms (`module_small_step_em.F:1102-1119`, `module_small_step_em.F:1341-1345`); ICON4Py and MPAS show a vertical-implicit path with explicit damping and divergence controls; Dinosaur/Pace provide third-path inspiration if neither local ADR converges. The output should be a manager decision with dissent, not another plan.

## §4 Diagnostic Sidecars Audit

Build order, cheapest-most-valuable first:

Estimated build: 2-4h. It extends the warm-bubble bound logic already in `scripts/m6_warm_bubble_test.py:72-78` and `scripts/m6_warm_bubble_test.py:275-304`.

2. **Sanitizer/finite-guard audit** (missed). Answers: is the replay "passing" only because `sanitize_state_with_stats` clipped or replaced values? Expected output: per-step candidate nonfinite, clip, changed counts, first bad candidate step. Estimated build: 2-3h because d02 replay already collects these fields (`src/gpuwrf/integration/d02_replay.py:313-322`, `src/gpuwrf/integration/d02_replay.py:468-487`).

3. **Limiter activation tracker** (missed). Answers: how often `_mu_continuity_increment` saturates and whether the limiter is hiding instability. This is critical because the limiter is a documented temporary stabilizer (`src/gpuwrf/dynamics/acoustic_wrf.py:456-475`).

4. **Field RMSE timeline** (manager named). Answers: which variables diverge first versus Gen2. Expected output: RMSE, bias, max absolute error for T2/U10/V10/qv2/w/theta by lead time.
